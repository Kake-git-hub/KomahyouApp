# 確定仕様：横断保証(INV)台帳（正本）

> オーナー承認済み（2026-07-11 制度化合意）。台帳の管理者＝**spec-curator**。
> 追加・文言改定は**オーナー承認**を要する。上限 15〜20 件。

## 位置づけ（この台帳は何か）

この台帳は、`spec-index.md` の 12 ドメイン正本（教室権限／保存・復元／振替／講習／…）と**直交する
「横断保証（INV = Invariant）」の層**である。ドメイン正本が「この機能はどう動くべきか（機能仕様）」を
定義するのに対し、本台帳は「**経路や実装が変わっても崩れてはいけないユーザー体験の保証**」を、
再現経路でなく**保証単位**で 1 行に固定する。

同じユーザー体験のバグが、経路（トリガ）を変えて繰り返し再発してきた——という実績が制度化の理由である。
例：講師日程表の二重表示は「生徒を別講師の机へ移動」（v1.5.388）で直した後、「講師 D&D の入替（swap）」
（v1.5.436）という別経路で再発した。バグを**再現経路でなく保証（INV-01）単位で固定**していれば、
2 件目の経路も同じマトリクスの空セルとして先回りで塞げる。

### 設計原則（この台帳の効き方）

- **バグを減らすのは INV の数ではない。** 効くのは「**テストで強制された INV**」×「**その保証の事故頻度**」。
  書いただけ（テストで守っていない）の INV はゼロ円。
- **効果は強く逓減する。** 最初の 10 件ほどで痛みの 8〜9 割を抑える。件数を増やすほど限界効果は小さい。
- **正しいスケールは横でなく縦。** INV の**数を増やす（横）**のではなく、各 INV の**操作マトリクスの網羅と、
  帰属・消化などの判定を担う権威関数への一元化（縦）**を深める。
- **守備範囲は「回帰」と「経路違い再発」のみ。** 性能・UX の分かりにくさ・新機能の仕様漏れは本台帳の対象外で、
  **spec-curator ＋ 実機確認**の仕事である。

### 重複ゼロ原則（ドリフト防止）

- **保証文は本書（`docs/spec-invariants.md`）に 1 回だけ書く。**
- ドメイン正本（`docs/spec-*.md`）と `memory/` は、保証文を複製せず **INV-ID で逆参照**する。
- 3 箇所へ同じ保証文を複製することは、`6793374`→`2dce7b4` 型の**巻き戻りドリフトの温床**として**禁止**する
  （元の同一性補完が別マージの spread 単純化で消えて再発した実例。§ INV-05）。

## 運用ルール（正式）

### 1. 台帳の管理

- 正本は本書。**上限 15〜20 件**（横に増やさない設計原則の運用上の歯止め）。
- 台帳への**追加条件は次の 2 つのみ**：
  1. **実際にユーザー体験のバグが起きた**（同種の再発を保証単位で固定したい）。
  2. **spec-curator が新機能でリスクの高い面を特定した**（着手前の予防的追加）。
- 追加・文言改定の権限は **spec-curator**（オーナー承認のもと）。個々の実装者が勝手に文言を薄めない。

### 2. 2 層制（強制 INV と準観察 INV）

- **強制 INV**（INV-01 / 02 / 03 / 06 / 07 / 08 / 09）：**操作マトリクステスト 1 ファイルを義務**とする。
  - ファイル名に INV-ID を含め、拡張子は `*.matrix.test.ts`（例：`inv01-teacher-attribution.matrix.test.ts`）。
  - 「経路違いの再発を 2 回以上、実績として確認済み」の INV が強制。
- **準観察 INV**（INV-04 / 05 / 10 / 11）：**非強制**。既存の経路テストで守るにとどめる。
  - **two-strike で昇格**：経路違いの再発の**2 件目**が出た時点で強制へ格上げし、マトリクスを新設する。
- 仕様が未決のセル（現実装と裁定が乖離しているマス等）は **`it.todo` で可視化**し、CI は緑を維持する。

### 3. 修正フローの完了定義（UX 系バグに必須・内部リファクタ／文言のみは免除）

UX に影響するバグを直したら、以下 4 点を満たして初めて「完了」とする：

1. **どの INV 違反かを特定**する。該当 INV が台帳に無ければ、**台帳追加が先**（§1 の追加条件）。
2. **経路テスト＋該当 INV のマトリクス拡張**（今回の経路のセルを埋める／薄い列は再確認する）。
3. **兄弟監査**：該当 INV のマトリクス行を読み下ろし、**隣接操作**（移動⇄入替⇄削除／生徒⇄講師／
   盤面⇄日程表／全ルールの列挙）に**穴が無いか**確認する。1 経路だけ塞いで隣で再発、を防ぐ。
4. **CHANGELOG・コミットメッセージに INV-ID を記載**する（違反履歴が台帳へ蓄積される）。

### 4. 分類ルール（症状でなく真因構造で分類）

- 各バグに **primary INV を 1 つ**割り当てる（必要なら secondary を追加）。
- **症状でなく真因構造で分類**する。例：
  - 同一操作が複数回**再実行**される → **INV-03**。
  - 別処理が状態を**上書き**する → **INV-02**。
  - **警告・割振対象判定**が実態と食い違う → **INV-09**。

### 5. CI（ブロッキング・改定ガード）

- マトリクステストは**毎 push ブロッキング**（既存 unit スイートに相乗り。`ci-tests.yml` の `unit` ジョブ）。
- **改定ガード（`inv-guard` ジョブ・実装済み）**：`*.matrix.test.ts(x)` を変更する差分は、**同一 push 内で
  `docs/spec-invariants.md` の改定を伴わなければ CI で落とす**。マトリクスの assert 薄化（＝ INV の格下げ）を、
  台帳の追認なしに通さないためのガード（`.github/workflows/ci-tests.yml`）。

### 6. regression-reviewer ＝ INV 監査ゲート（4 点出力）

リファクタ・マージ衝突解決の後などに、`regression-reviewer` が読み取り専用で以下 4 点を出力する：

1. **差分が触れる INV の特定**＋コミットの INV-ID 記載チェック。
2. **兄弟被覆の検証**（隣接操作に穴が無いか）。
3. **INV 格下げの検知**（マトリクスの assert が薄まっていないか）。
4. **権威関数の分散検知**（帰属・消化などの判定が複数箇所に散っていないか）。

### 7. 権威関数の一元化（別タスク・マトリクス先行）

- 講師帰属の判定（`resolveDeskTeacherIdentity` 等の一元化）は**別タスク**として中期に実施する。
- **マトリクスを先に整える**（INV-01 マトリクスに守られた最初の改修として着手する）。これにより、
  一元化リファクタが帰属一意（INV-01）を巻き戻さないことをマトリクスが保証する。

## INV 各論

各 INV は **保証文 ／ 層 ／ 仕様上の例外・確定仕様 ／ 違反履歴 ／ マトリクステストファイル ／ 担保状況**
を持つ。違反履歴の版数・commit・日付は、統合バグ台帳から**転記**したもの（捏造せず、根拠のある版のみ）。
