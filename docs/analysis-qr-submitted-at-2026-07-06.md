# 解析報告: 2026-07-06 15:25 講習巻き戻り障害 × QR提出時刻の突き合わせ

目的: v1.5.396 で修正した「講習自動割振の巻き戻り」について、本番データで真因説を検証し、
対策が報告された障害に対して有効かを判定する。

## 結論（要旨）

1. **障害教室は日大前校で確定。** saveAttempts に報告どおりの「割振→手動移動→数分後リセット」が
   バイトサイズ増減として記録されていた（下表）。緑が丘校は同時刻に急増減なし・薬円台校は保存ゼロ。
2. **A4（QRトークン発行競合）は 15:25 障害の発火経路ではない。**
   A4 の窓は「新規トークン発行の await 中」しか開かないが、**7/6 はどの教室でも発行ゼロ**
   （lectureSubmissions の createdAt 直近: 日大前 6/30・緑が丘 6/27・薬円台 6/5。
   削除経路 `deleteLectureSubmissionDoc` は呼び出し元ゼロの死蔵で、doc が消えた可能性もない）。
   さらに全履歴クロス照合（発行±10秒に別人の提出）でも該当 **0件**（3教室・計2,266 docs）。
3. **最有力経路は A3（stale 書き戻し・commit `a0dba9f`）**（2026-07-07 追記で確定度更新）。
   オーナー経由で被害生徒が特定された: **15:18:28 に提出した生徒U（中3）本人**であり、
   提出の取り消しはしていない。初回評価誤除去（`9bafd01`）が外せるのは未提出の生徒の配置
   だけなので、**提出済み生徒の配置が消えた本件は初回評価誤除去では説明できない**。
   一方 A3 は「盤面(weeks)だけが古い状態へ戻り、提出(subjectSlots)は残るため講習残数が
   満数へ戻る」不具合で、提出済み生徒でも巻き戻る＝本件の症状と一致する。
   さらにリセット保存 v1615（6,004,606 bytes）は、**生徒Uの提出到着(15:18:28)の1秒前**の
   状態である v1611（15:18:27・6,004,605 bytes）とほぼ同一サイズ（差1バイト）で、
   「提出→割振→移動の約5分間がそっくり古い状態で上書きされた」という A3 の署名に合致する
   （完全一致でないのは、生き残った提出データの再反映等の微小差で説明可能）。
   初回評価誤除去は、同時に割り振られていた未提出生徒の配置消失に寄与した可能性は残る。
4. **判定: 対策は有効と評価できる（条件付き）。** 症状に合致する経路（初回評価誤除去・A3・A4）は
   3つとも v1.5.396 で修正済み・本番稼働中（確認時ライブ v1.5.402）・回帰テスト 247件グリーン
   （App.test.ts / pendingSnapshotVersionGuard.test.ts / ScheduleBoardScreen.test.ts）。
   ただし v1.5.396 の本番反映は 7/6 21:36 JST で、日大前校の当日操作終了（最終保存 18:10）より後。
   **修正後に同教室で講習自動割振はまだ行われておらず、「再発ゼロ」の実地実証は今後の稼働待ち。**
