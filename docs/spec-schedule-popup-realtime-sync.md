# 確定仕様：生徒/講師日程表のリアルタイム同期（自動同期の復活）— 正本

> 2026-07-08 オーナーQ&A（10問中4問＋共通2問）で確定。未実装（実装タスクは
> `docs/handoff-popup-sync-and-dnd.md` 参照）。
>
> ⚠️ **これは `spec-index.md` 大方針6「日程表は追従同期をやめ『最新表示』ボタンに」の正式な上書き**。
> 旧決定（2026-06-05・メモリ軽量化のため常時同期停止 / commit `bff131c`・`b51c49d`）は
> 「毎編集で全量ペイロードを二重生成する実装」を止めたものであり、本仕様は
> **負荷対策（デバウンス＋変化検知＋表示範囲限定）を前提に自動同期だけを復活させる**。
> 旧障害の原因構造（下記 §D）を復活させてはいけない。

## A. 背景と目的

- 現状、日程表ポップアップ（生徒/講師）は「最新表示」ボタンを押した時だけ盤面を反映する。
- ユーザー（教室スタッフ）はリアルタイム同期だと思い込んでおり、「日程表に反映されていない」
  という問題報告・問い合わせが繰り返し発生している。
- 盤面の変更が自動で日程表に映るようにし、勘違い起因の報告をなくす。
  **ただしアプリを重くしない（旧障害を再発させない）ことが絶対条件**。

## B. 確定事項（オーナーQ&A 2026-07-08）

| # | 論点 | 決定 |
|---|------|------|
| 1 | 同期タイミング | **盤面編集後のデバウンス自動同期（1〜2秒）**。連続編集中は送らず、手が止まったら1回だけ同期。編集ごと即時・保存時のみは不採用。 |
| 2 | 負荷対策 | **送信側の変化検知＋表示範囲限定生成**。変化がなければペイロードを生成すらしない。生成はポップアップで選択中の期間・絞り込み分のみ（現行の range 機構を踏襲）。ポップアップ側の受信フィンガープリント比較（既存 `buildPayloadFingerprint`）による再描画スキップも維持。 |
| 3 | 「最新表示」ボタン | **「期間・生徒名適用」（講師日程は「期間・講師名適用」）に改名して残す**。同期は自動に一本化されたことをUIで明示し、ボタンは期間・絞り込みの適用専用とする（手動同期の保険も兼ねる）。 |
| 4 | 対象ポップアップ | **生徒日程＋講師日程のみ**。全体日程（全生徒/全講師）・講習（特別期間）ポップアップは対象外（従来どおり開いた時＋明示操作時のみ）。 |
| 5 | 保存アーキテクチャ | 変更なし。自動同期は**同一端末内の「ローカル盤面状態 → ポップアップ」の反映**であり、多端末間同期ではない（1教室1端末方針は不変）。保存は従来どおり手動保存ボタン → `saveClassroomSnapshot`。 |
| 6 | 展開 | **featureRollout（`development-only` → 実機検証後 `all-classrooms`）**。盤面D&D（v1.5.338）と同じ手順。 |

## C. 挙動仕様

1. 生徒日程/講師日程ポップアップが開いている間、盤面（`boardState`）に変更が起きたら、
   最後の変更から **約1.5秒（1〜2秒で実装時に調整可）** 後に該当ポップアップへ
   `schedule-data-update` を送る。
2. デバウンス窓の間に何回編集しても送信は1回。ポップアップが閉じていれば何もしない
   （タイマーも発火時にポップアップ有無を再確認して破棄）。
3. 送信側で盤面変更の有無を軽量に判定（リビジョンカウンタ方式を推奨）し、
   **前回送信時から変化がなければペイロード生成・送信ともに行わない**。
4. ペイロードはポップアップが現在選択している期間・生徒/講師絞り込みの範囲のみ生成
   （現行の `syncStudentSchedulePopup` / `syncTeacherSchedulePopup` の range 機構をそのまま使う）。
5. 「最新表示」ボタンは「期間・生徒名適用」/「期間・講師名適用」に改名。
   機能（範囲・絞り込み適用＋最新盤面取り込み）は従来どおり維持する。
6. ポップアップを開いた時の初期同期（`schedule-popup-ready`）、明示リフレッシュ
   （`schedule-refresh-request`）、範囲適用（`schedule-range-update`）の既存3経路は変更しない。

## D. 旧障害の原因構造と再発防止（禁止事項）

2026-06-05 のメモリ障害（日程表を開いたまま出席を連続編集すると heap が増え続ける）の真因は
以下の複合だった（commit `bff131c` / `b51c49d` に記録）：

1. **二重同期**：App（`App.tsx` の `syncStudentSchedulePopup` 等）と盤面
   （`ScheduleBoardScreen.tsx` の直呼び effect）が同一ポップアップを別々に再同期していた。
2. **毎編集の全量再生成**：effect の deps に `boardState.weeks` 等の編集で identity が変わる値が
   入っており、出席1クリックごとに全生徒・全講師分の HTML ペイロードを再生成していた。
3. **localStorage 全教室ミラー**：自動保存ごとに全教室を `JSON.stringify` → quota 例外 → 破棄の
   巨大文字列生成（無効化済み）。

これを踏まえた**禁止事項（回帰ガード）**：

- `App.tsx` の `syncStudentSchedulePopup` / `syncTeacherSchedulePopup` の
  **`if (!force) return` ゲートを外さない**。自動同期は force ゲートを迂回する別経路ではなく、
  デバウンス済みトリガから force=true で明示的に1系統だけ呼ぶ。
- `ScheduleBoardScreen.tsx` の既存2つの同期 effect（deps `[scheduleSyncTrigger]` のみ）の
  **deps に boardState 系の値を戻さない**。
- **App と Board の両方から自動同期を発火させない**（自動経路のオーナーは1箇所に定める）。
- localStorage ミラー（`writeWorkspaceToLocalStorageSync` の常時書き込み）を復活させない。
- デバウンスなしの即時同期にしない（Q&A で明示的に不採用）。

## E. 受け入れ条件

1. 生徒・講師の両ポップアップを開いたまま盤面で出席・配置を連続 20 回以上編集しても、
   `schedule-data-update` の送信回数が編集回数に比例しない（デバウンスで数回に集約される）こと。
2. 同状況で heap（メモリ）が編集のたびに単調増加しないこと（staging 実機で
   `?memlog` / `bumpMemCounter` カウンタと DevTools メモリ計測により確認。
   commit `bff131c` が宿題とした「両 popup 開いたまま連続出席入力時の heap 再測定」を兼ねる）。
3. 盤面編集の停止から約2秒以内にポップアップへ変更が反映されること。
4. 盤面に変化がない場合（同一状態への再レンダー等）はペイロード生成・送信が行われないこと。
5. 「期間・生徒名適用」ボタンの絞り込み・期間適用が従来どおり動くこと。
6. ポップアップ非表示時に自動同期処理が走らないこと。
7. featureRollout 無効教室（`development-only` 段階の本番3教室）では従来挙動のままであること。

## F. メリット・デメリット

**メリット**

- 「日程表に反映されない」という勘違い起因の問題報告・問い合わせがなくなる。
- 盤面調整→日程表確認のワークフローがボタン操作なしで回る（生徒日程表D&D機能と相乗）。
- 旧実装と違い負荷が編集回数に比例しないため、大教室でも安全。

**デメリット・リスク**

- デバウンス1回分（〜2秒）の生成コストは残る。生徒数の多い教室で編集が非常に頻繁な場合、
  数秒おきのペイロード生成が発生する（→ 受け入れ条件2で実測確認）。
- 「自動で映る」への期待が確立するため、万一自動同期が不発の場面（例外・タイマー破棄漏れ）が
  あると旧来より不信感が強い。改名後のボタンが手動の保険として残るのが救済手段。
- 実装が盤面編集経路に触れるため、既存の force ゲート・同期 effect を壊すと旧障害が再発する
  （→ §D の禁止事項と回帰テストで防ぐ）。

## G. 本番環境への影響範囲

- **データモデル・Firestore・Cloud Functions への変更なし**。保存経路（`saveClassroomSnapshot`）不変。
  本番データへの新たな書き込みは発生しない。
- 触るのはフロントのみ：`src/App.tsx`（自動同期トリガ）、
  `src/components/schedule-board/ScheduleBoardScreen.tsx`（盤面変更の検知）、
  `src/utils/scheduleHtml.ts`（ボタン改名）、`src/utils/featureRollout.ts`（キー追加）。
- featureRollout により本番3教室は昇格コミットまで挙動不変。ただし**ボタン改名だけは
  rollout ゲート外（全教室即時）にするか検討が必要** — 自動同期が無効な教室で改名だけ先行すると
  「最新表示」の意味が失われるため、**改名も featureRollout で出し分ける**（無効教室は「最新表示」のまま）。
- 旧障害再発時の影響は「アプリが重くなる」（データ破損はしない）。ロールバックは Hosting 巻き戻し
  （`docs/runbooks/rollback.md` A）で即時可能。
