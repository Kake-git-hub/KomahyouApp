# 外部DB / 認証候補メモ

現状のコマ表アプリは Vite + React + TypeScript の純フロント構成で、利用者も開発者と教室管理者に限定されています。これを踏まえると、初期導入は「認証と DB を同じサービスで持てる」「行レベル権限制御ができる」「フロントから段階移行しやすい」が重要です。

## 候補比較

### 1. Firebase Authentication + Firestore

- 向いている理由
  - 今回のように Firebase を採用決定した場合、認証と DB を同系統でまとめられる。
  - フロント主体の構成へ載せやすい。
  - 少人数運用であればセットアップが比較的速い。
- デメリット
  - 現在のデータは結合や集計が多く、Firestore のドキュメント設計に寄せると長期的に複雑化しやすい。
  - Auth ユーザー発行やロール管理を安全に回すには Cloud Functions か運用スクリプトが欲しい。
  - SQL ベースの厳密な正規化よりは、snapshot ベース移行の方が現実的になる。

### 2. Supabase

- 向いている理由
  - PostgreSQL ベースで、将来の正規化設計にそのまま乗せ替えやすい。
  - Email/Password 認証をすぐ載せられる。
  - RLS で「開発者は全教室」「管理者は担当教室のみ」を DB 側で制御できる。
  - Vite フロントから段階導入しやすく、今回の実装でも採用しやすい。
- デメリット
  - `auth.users` の新規発行を安全に自動化するには Edge Function か別 API が必要。
  - 複雑な帳票・大量バッチを将来載せるなら設計責任はアプリ側に残る。

### 3. Neon + Clerk

- 向いている理由
  - PostgreSQL とモダン認証の組み合わせで、将来の Web SaaS 形に寄せやすい。
  - UI を含めた認証体験は整えやすい。
- デメリット
  - DB と認証が別サービスなので、初期セットアップと権限境界が少し重い。
  - 今のアプリ規模ではやや過剰。
  - フロントだけで閉じず、BFF か API 層を用意したくなりやすい。

### 4. AWS Amplify + Cognito + RDS

- 向いている理由
  - 将来の多拠点展開や厳格な運用要件には強い。
  - IAM を含めた統制が取りやすい。
- デメリット
  - 今の 2 ロール構成には明らかに重い。
  - 初期構築、運用、デバッグの学習コストが高い。
  - UI 試行を速く回したい現フェーズとは合いにくい。

## 現在の判断

比較上は Supabase にも強みがありますが、このリポジトリでは Firebase を使う前提で進めています。

- 認証は Firebase Authentication
- 外部保存は Firestore
- 保存単位は教室ごとの snapshot
- 管理者アカウント発行は Cloud Functions か管理用スクリプトを別途用意する前提
