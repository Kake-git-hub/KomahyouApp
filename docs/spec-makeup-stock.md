## 1. 発生（積まれる）

### A. 休日振替（自動・★残す）
- **日付をクリックして休日（holidayDates）にすると、その日に通常授業があった生徒は全員「休日振替」として未消化に自動計上**される（理由ラベル `休日振替`/`定休日振替`）。
- ★**計上タイミングは過去・未来を問わず、休日設定した時点で即時**（旧実装は「その日が過ぎてから」だったが、即時計上に変更）。
- 休校曜日（closedWeekdays）由来の潰れた通常授業も同様に振替計上。`forceOpenDates` は除外。
- 集計単位は **生徒 × 科目**。
- ★**即時計上の適用範囲（2026-07-04 確定・C1）**：即時計上は**現行テンプレの有効期間内（テンプレ凍結日 `templateFreezeBeforeDate` 以降・当年度 `schoolYear`）の休日に限る**。テンプレ凍結より前・過去年度の休日振替は**自動計上の対象外**で、必要なら**手動追加で復帰**する。
  - 理由：在庫計算 `computeAutomaticShortageOrigins` は**現行 `regularLessons`（＝凍結後テンプレ由来 rows）だけ**を走査し、当年度以外を除外する。凍結で通常授業行の開始日が凍結日に更新されると、凍結前の休日はその row の在籍期間外＝origin 未生成（autoShortage=0）。**在庫計算は履歴合成しない**（schedule 表示は `buildCombinedRegularLessonsFromHistory` で凍結前も補うが、makeup 在庫は補わない）。
  - この非対称は**本番事故の真因**：テンプレ凍結後、凍結前の 7/20（海の日）休日振替が4名分だけ在庫から消失した（memory `komahyou-occupied-origin-and-suppressed-makeup` / `komahyou-handoff-restore-720`）。**「凍結前は手動追加で復帰」を正式な回復手段とする**（在庫計算の履歴合成化は全生徒の凍結前振替が大量復活し数量激変＝危険なため採らない）。
- ★**初期設定日フロア（2026-07-04 確定・C4）**：**初期設定完了日（`initialSetupCompletedAt`）より前**の休日は自動計上せず、運用開始前の残は**繰越（§1-C・初期設定の未消化振替）で登録**する。繰越と自動計上の境界を固定し、初期設定日を動かす変更で過去分が大量増減しないようにする。

### B. 欠席登録（明示・★実装済み）
- 盤面の通常コマの生徒を **「欠席」登録すると、その生徒×科目の振替ストックに1コマ積まれる**（休日以外の個別欠席用）。
- 欠席登録した**コマは盤面から消さず「欠席」表示で残す**。席は**ロックせず、クリックで別生徒・体験・メモを追加できる**。
- 振替元として日付・時限を記録。理由ラベルは持たない。
- ※**2026-06-08 精査で実装済みと確認**。生徒メニュー「休み」ボタン（`handleMarkStudentAbsent`）が本要件を満たす。振替なしの欠席は別ボタン「振無休」（absent-no-makeup）。詳細は `docs/SESSION-HANDOFF.md` §0。

### C. 繰越
- 運用開始時の繰越は、初期設定の「未消化振替」で登録した分が最初から積まれる（`docs/spec-save-restore.md` §6）。

### D. 同時間帯重複振替（自動・2026-07-04 追加・C4）
- **同一日・同一時限に同一生徒が複数コマに乗る構成**が生じると、余剰分を「同時間帯の重複」として振替に自動計上する（理由ラベル `同時間帯の重複`・`computeScheduleConflictOrigins`）。
- ★**計上タイミングは過去分のみ**（休日振替が過去/未来即時なのと**非対称**）。初期設定日フロア（§1-A の `initialSetupCompletedAt` 未満はスキップ）は重複にも適用。

## 2. 消化／戻し

- ストックを選んで**空きコマへ置くと、残が1減る。**
- 置いた振替を**盤面から外す（ストックへ戻す）と、残が1増える。**

### 消化アルゴリズム（2026-07-04 明文化・B5）★消してはならないガード

- 消化は**振替元日付単位で突き合わせる**：盤面に置いた振替が持つ元日付（`makeupSourceDate`）を残 origin から除き、元日付を持たない配置分は**古い origin から順に消す**（`consumeOriginDates`）。
- **未消化振替一覧で選んだ振替元日付を配置に引き継ぐ**（最古へ巻き戻さない・`resolveSelectedMakeupOrigin`）。振替の発生源（休日/欠席/繰越/重複）が表示ラベルに直結するため、消化順は室長の見え方に影響する。
- ★**残数は消化済みを除いた origin 件数に一本化する**（`balance = Math.max(0, remainingOriginDates.length - 手動独立分)`）。**過去に「振替1コマ置くと残が2件減る（3→1）」過剰減算バグ**（`overAssignedRegularLessons` の減算が消化と二重にかかった）が起きたため、balance からの overAssigned 減算は撤去済み（§4）。回帰テスト `makeupStock.test.ts`（3→2）で保護。**この消化ロジックを「単純化」で消すと再発する。**

### 削除（抑制）／復帰（2026-07-04 明文化・C3・B1）

- 未消化振替一覧からの**削除**、または盤面の通常授業/振替コマの**削除**は、`suppressedMakeupOrigins`（生徒×科目キー→抑制した振替元日付）に記録され、在庫計算は自動 origin・手動調整の和集合から**抑制分を差し引いて**残数を出す。
- ★**削除は元に戻せない（現状は undo 以外に復帰経路が無い）**。この復帰UIの欠如は、本番で「本物の休日振替（7/20・4名）が偽 origin の一括削除に巻き込まれて消え、戻せなかった」事故の中核だった（memory `komahyou-occupied-origin-and-suppressed-makeup`）。
- **復帰UI（削除した未消化振替を戻す）を追加する**（**Issue #39**）。実装までは「削除は原則やり直せない・誤削除は undo で戻す」運用とする。

### 「空にする」での振替再出現 ／ 登録解除で振替は戻らない（オーナー確定 2026-07-06・現状のまま）

以下の2つの非対称は、いずれも 2026-07-06 にオーナーが「**現状の挙動が正**」と確定した（確定に格上げ）。

- **「空にする」（日付一括 `handleClearStudentsOnDate`）はその日に置いてあった振替を在庫に再出現させる**。
  「削除（抑制）」と違って `suppressedMakeupOrigins` に積まないただの**単純クリア**なので、盤面から消えた振替の
  origin は在庫計算（自動 origin ∪ 手動調整 − 抑制）に**再び現れる**。これは意図した現状仕様。
  - ★**講習との対比（台帳方式ゆえ再出現しない）**: 講習は盤面を走査せずデルタ台帳で残数管理するため、
    「空にする」で日付を空にしても在庫（希望数）が**再出現しない**（`spec-lecture-stock.md` §4-2 の「空にする＝
    希望数もストックも触らない単純クリア」）。振替＝盤面走査／講習＝台帳、というモデル差がこの非対称の根。
- **登録解除では振替は在庫へ戻らない（振替は登録解除に連動しない）**。生徒の登録解除で盤面から自動除去するのは
  講習（special）だけで、自動割当されていた振替（makeup）は盤面に残す（未消化振替へ戻さない）。
  この非対称は意図的な確定仕様。正本は `spec-special-session-submission.md` E-2b ①、
  回帰固定は `ScheduleBoardScreen.test.ts › removeStudentAssignmentsFromSpecialSession`。

### 自動割振での振替同時割当（2026-07-04 明文化・B7）

- 消化は手動配置のほか、**講習の自動割振モーダルの「未消化振替も同時に自動割り当てする」（既定 OFF・2026-06-27 確定）** でもできる（規則詳細は `spec-auto-assign-rules.md`）。講習と同一規則・スコアを共有する共有コアで、末尾に振るい順（古い振替元優先）を評価する（`findBestAutoAssignCandidate` / `buildMakeupAutoAssignPendingItems`・v1.5.331 本番反映）。
- ★**balance を超えて生成しない**（`buildMakeupAutoAssignPendingItems` は残数を上限に pending を展開）。盤面再計算ベースの残数を超えて置かない保証で、**消してはならないガード**。

## 3. 表示・ルール（現行から維持）

- 集計単位は **生徒 × 科目。**
- 振替ストックはコマ表操作後も**開いたまま維持**する。
- **手動追加した生徒は振替ストックにカウントしない。**
  - ★正確な意味（2026-07-04 明文化・B4）：**手動追加した授業（`manualAdded`）を消化/計上に数えない**。繰越・欠席で積んだ未管理生徒の残数は出る。
- 振替中／移動中に**同じコマへ同じ生徒が既にいると配置を拒否**し、選択状態は維持する。

### 集計キーと未管理生徒の扱い（2026-07-04 明文化・B4）

- キー＝`生徒キー__科目`。**生徒キーは管理生徒＝生徒ID、未管理/手動追加＝`name:表示名`（内部的に `manual:` プレフィックス経由の場合あり）**。
- **未管理生徒の表示名・科目は `fallbackMakeupStudents` に退避**し、生徒が盤面から消えても一覧に名前を出せる。
- 生徒が後から管理登録されると、`normalizeManagedMakeupStockKeyByIdSet` が `manual:`／`name:` キーを管理IDへ**再吸着して合算**する（別々に二重計上しない）。

## 4. 廃止する概念

- ★**マイナス残（希望回数を超えて配置したときの過剰配置表示）は廃止**。残数は 0 を下限とし、マイナスにしない。
  - ★正確な意味（2026-07-04 確定・C2）：撤去したのは **balance への `overAssignedRegularLessons` 減算** である。フィールド自体はデータ辞書として残る（下記）。
  - **データ辞書**：`overAssignedRegularLessons`（`MakeupStockEntry` の**参考値・balance に一切影響しない**）／`negativeReason`（**常に `null`** の後方互換の残骸）。将来フィールドごと撤去するかは別途（機能は満たされている）。
- ~~autoShortage の自動計算~~ → **残す**（§1-A）。

## 5. データモデル（2026-07-04 明文化・B2）★消してはならないガード

未消化振替の残数は、次の3系統の origin の**和集合から抑制分を差し引いて**算出する（`buildMakeupStockEntries`）:

1. **自動 origin**：休日/定休日振替（§1-A・`computeAutomaticShortageOrigins`）＋同時間帯重複（§1-D・`computeScheduleConflictOrigins`）。現行テンプレ・当年度・初期設定日フロア以降のみ。
2. **手動調整 `manualMakeupAdjustments`**（生徒×科目キー→振替元日付の配列）：**繰越（初期設定 `reasonLabel:'初期設定'`）・欠席（§1-B）・盤面からの戻し（§2）の3経路を1つのマップに混載**する。区別は各 origin の `reasonLabel` の有無のみ。
3. **抑制 `suppressedMakeupOrigins`**（生徒×科目キー→抑制した振替元日付）：削除で積まれ、残数計算で**差し引かれる**（§2 削除）。

- 残数 ＝（1 の自動 origin ∪ 2 の手動調整）− 3 の抑制、を消化（§2 消化アルゴリズム）で消した残り。
- ★このデータモデル（3系統の混載と抑制の差し引き）を**消化・相殺・抑制のいずれかを触る変更で崩すと、二重計上／消失が起きる**。過去の過剰減算バグ（§2）・本番の振替消失事故（§1-A）はいずれもこの構造の未定義が温床だった。**ライフサイクル（誰が積み・誰が消すか）を固定して据え置く。**

## ★ 振替と講習の非対称（2026-07-04 明文化・B8）
