## A. 反映日（effectiveStartDate）の境界

**Q1.** ✅ 確定。テンプレ反映日**より前**のコマ表は、テンプレ反映で**一切変更しない**（講師・生徒・出欠・メモすべて不変）。再構築は反映日**以降**のみ。
　根拠: L3068「禁忌: 反映日以前のコマ表は不変」, L3051。

**Q2.** ✅ 確定。反映日**以降**に既にあった休日(holidayDates)は**削除されて営業日に戻り**、テンプレ授業が入る。反映日**より前**の休日は残す。
　根拠: `filterTemplateOverwriteHolidayDates`（`< effectiveStartDate` だけ残す, L1049-1051）。

**Q3.** ✅ 確定。反映日 = `templateFreezeBeforeDate` に設定され、これより前は「凍結（preTemplate基準＝旧データ）」として扱う。

## B. 休日・定休日・営業日との力関係（★非対称・要注意）

**Q4.** ✅ 確定（＋★実装デルタあり）。テンプレが消すのは **holidayDates（手動の個別休日）だけ**。**closedWeekdays（定休日＝日曜）と forceOpenDates（強制営業日）はテンプレでも消さない**。
　＝「テンプレ < 定休日」「個別休日 < テンプレ」。定休日にテンプレ授業を置いても定休日が優先（振替化）。
　**★追加要件**：混乱防止のため、**テンプレ編集モードでは定休日(closedWeekdays)のコマに講師・生徒を配置できないようにする**（定休日セルは配置不可＝編集ブロック）。

**Q5.** ✅ 確定。テンプレ反映**後に**、反映日以降の日へ**新たに手で休日を設定**した場合、その休日は有効（その日だけテンプレ授業が振替化）。＝「反映時に既存休日はクリアするが、反映後に付けた休日は生きる」。

## C. 反映日以降の再構築でストックがどうなるか（overwrite時）

**Q6.** ✅ 確定。反映日以降の全セルを一旦まっさらにクリア（講師・手動講師・自動割振由来・メモ・出欠status・授業すべて消去）してからテンプレ通常授業を再配置。
　根拠: L3053-3065。

**Q7.** ✅ 確定。反映日以降に**配置済みだった振替(makeup)・講習(lecture)**は、消す前に**未消化ストックへ返す**（「○件をストックへ戻しました」表示）。

**Q8.** ✅ 確定。反映日以降に**欠席で生まれていた未消化振替/講習**は、再構築で通常授業が戻るので**相殺（取り消し）**する（二重計上防止）。
　根拠: L3001-3019。

**Q9.** ✅ 確定。反映日以降の**手動追加した振替(makeup, manualAdded)**は**復元しない**（テンプレ再生成と二重になるため）。占有・休日由来の本来のショート分は再計算で自動復活。
　根拠: L2982-2998。

**Q10.** ✅ 確定。反映日以降ぶんの 手動振替調整・各種抑制(suppressed)・希望回数補正(scheduleCountAdjustments) は**クリア**、反映日より前ぶんは**保持**。
　根拠: L2948-2952, 3030-3047。

## D. テンプレ展開（何を・どこまで生成するか）

**Q11.** ✅ 確定（★実装デルタあり）。**複数年一括生成・上限年度(2031)は撤廃**。テンプレは**反映日からその年度末(3/31)まで**の通常授業だけを生成する。
　理由: 長期生成は不要で上限を決める必要もない。**年度替わりでは必ずテンプレも変わる**運用のため、単年度生成で問題ない。
　現状: `regularLessonTemplate.ts` L262/271 が `maxSchoolYear = 2031` まで多年度ループ → **単年度（effectiveSchoolYear のみ）に変更**。

**Q12.** ✅ 確定。授業期間は「反映日 or 年度開始(4/1)の遅い方」〜「年度末(3/31)」。（Q11 の単年度化後は実質「反映日〜当年度3/31」）。

**Q13.** ✅ 確定。科目は生徒の**生年月日×その年度基準日**で表示科目に自動解決（学齢で算↔数など切替）。基準日＝その年度の共有開始日。

**Q14.** ✅ 確定（★実装デルタあり）。テンプレ盤面に出す対象は **「入会前」＋「在籍中」** の生徒（入会前の生徒もテンプレに配置できるようにする）。**講師も同様**（入会前＝勤務開始前も含める）。退会後・学齢外は除外。
　現状: `filterTemplateParticipantsForReferenceDate`（L209 / 呼び出し L2849, L5519）は referenceDate 時点の**在籍のみ** → **入会前(未来開始)も含める**よう条件緩和。

## E. 高3卒業・年度跨ぎ・世代交代

**Q15.** ✅ 確定。高3など年度途中・年度末で卒業/退会する生徒は、在籍・学齢判定で**自動的に外れる**（手動操作不要）。**高3は3月最終日(3/31)までを在籍**とする。
　（Q11 で単年度生成になったため、跨年度の卒業判定は次年度テンプレ作成時に自然に解決される。）

**Q16.** ✅ 確定（★実装デルタあり）。「世代交代」＝反映日より前は旧テンプレ/旧データのまま、反映日以降だけ新テンプレに置換。複数回変更すると反映日で区切られ履歴(history)が積まれる。
　**★追加要件**：**履歴(`regularLessonTemplateHistory`)の保管は最新3件まで**（古い側を破棄）。
　現状: L2920-2924 は `effectiveStartDate >= 新反映日` を除去して追加するだけで**件数上限なし** → **最新3件にトリム**。

## F. 保存モード・履歴・入出力

**Q17.** ✅ 確定（★実装方針）。ユーザー視点では保存モードを**分けない**（overwrite/定義のみ保存の区別を意識させない）。
　ただし内部的に区別することで**挙動が軽くなる等のメリットがある場合は、内部実装としての区別は残してよい**（UI上は1つの「保存」として見せる）。

**Q18.** ✅ 確定。履歴(history)は「新反映日 **以上** の旧履歴を捨て新テンプレを追加」。`preTemplateRegularLessons`（予定回数＝通常回数の基準）は**初回保存時のみ**退避し以降維持。
　根拠: L2920-2929。（件数上限は Q16 の3件を適用）

**Q19.** ✅ 確定。テンプレは**Excel出力/取り込み**可。取り込み時に開始日(反映日)をブック内から選んで適用。**取り込み時も上書き再構築扱い**。

## G. テンプレ編集中の操作

**Q20.** ✅ 確定。テンプレ編集モード中の 移動・削除・追加・詰め替え(pack)・Undo/Redo は**テンプレ下書きだけ**を編集し、**保存するまで本番コマ表に反映しない**。
