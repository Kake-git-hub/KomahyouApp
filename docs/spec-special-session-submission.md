# 確定仕様：特別講習データ・提出ページ（To-Be 正本）

> 全体見直しセッション（2026-06-07）でオーナー確認済みの「目標仕様」。

## A. 教室側：特別講習データ画面

- この画面で管理するのは **講習名と講習期間だけ**。
- 「新しい特別講習を作成」で講習名＋期間を登録、一覧から**編集・削除**できる。
- 新規教室の初期状態は**講習データ空**で開始（過去サンプルはロード時に除去）。
- ★**作成・編集のバリデーション（重複禁止）**（監査 A3・オーナー確定 2026-07-04）:
  - **講習名は重複不可**（同名の別講習を作れない）。
  - **講習期間は他講習と重複不可**（1日でも重なると作成/編集を拒否）。編集で重複した場合は**編集前の値に巻き戻す**。
  - 根拠: 提出トークンの科目/集団判定が「講習開始日基準の学年」に依存するため、期間重複を許すと同一日が複数講習に属し
    提出先が曖昧になる。この制約は設計判断として意図的（`validateSessionDraftAgainstSessions` / `findSessionDateRangeConflict`）。
- ★**欠席不可・希望の登録経路は「日程表での登録」に一本化する。**
  - 現行の「コマ表の講習期間帯をクリック→別タブで欠席不可入力」経路は**廃止**（別タブ生成器・handler・effect は撤去済み。
    編集パネル内に残っていた死んだ案内文も 2026-07-04 に日程表/QR 案内へ統一・監査 A1/C1）。
  - （注）詳細はブロック⑨（日程表）で確定。

## B. QR／提出リンクの配布・管理

- 生徒・講師それぞれに**個別の提出トークン（QRリンク）**を発行して配る。
- 室長は提出状況を**リアルタイムで受信**できる。
- ★**「ロック解除」ではなく「登録削除」で再提出可能化する**：
  - 室長がその人の登録を削除すると、提出がリセットされ、**同じQRで再提出できる状態**に戻る（後述 E）。
- ★**トークンの発行/更新と後追い反映**（監査 B6・オーナー確定 2026-07-04）:
  - トークンは**一度発行したら維持**（同じ QR を配り続けられる）。`ensureSubmissionTokens` は未発行の person にのみ採番（冪等）。
  - **盤面変更（青表示の occupiedSlots）・休日設定・限数（slotNumbers）・集団コマ・オプション文言（optionLabels）は
    既発行 QR にも後追い反映**され、提出ページが常に最新の盤面/休日を映す（`updateSubmissionOccupiedSlots` /
    中3の集団選択肢後埋め `updateSubmissionGroupClassEligibility`）。

## C. 提出ページ（生徒・講師 共通）：出席不可コマ

- リンクを開くと**本人用の提出フォーム**（講習名・氏名・期間を表示）。
- 表は「日付 × 限」。**出席できないコマをタップで×**。
  - **日付タップで終日不可／限ヘッダタップでその列を全部不可。**
- 既に盤面に組まれているコマは**青く表示**され、そこを不可にしようとすると確認が出る。
- **休校日は灰色で選択不可。**
- ★**スマホ最適化とデバッグ画面**（監査 B8・オーナー確定 2026-07-04）:
  - 提出ページは**スマホ最適化**（iOS/Android 別のビューポート幅・zoom 補正を初回ペイント前に適用。Android は表を逆ズームで全幅化）。
  - `#/submit-debug` は**実機表示調整用のダミー画面**（本番トークン不要・ネットワーク非使用）。表示崩れ調査時の入口。

## D. 提出ページ（生徒のみ）：希望科目数

- 科目ごとに**＋／−で希望コマ数**を入力できる（合計を表示）。
- 「**通常授業のみ（講習なし）**」チェックを付けると、講習ストック対象外になる（`docs/spec-lecture-stock.md` と整合）。

### D-1. 集団授業の提出（相互参照）（監査 B1・オーナー確定 2026-07-04）

- **集団授業の提出（`groupClassParticipation`・中3の集団理科/社会の参加可否・提出ページの集団列）の正本は
  `docs/spec-group-lesson.md` §C/§E**。本書はこれを相互参照する（提出ページには中3かつ集団科目がある場合のみ集団列が出る）。
- **保護者所有フィールド（`groupClassParticipation`・後述 D-2 の `optionChecks`）の反映は 2経路**（QR購読 ／ 生徒日程表の登録）。
  - 購読側の反映は **union（削減しない）**＝空 doc が既存の集団参加を消さない。**過去の本番消失事故
    （v1.5.336・緑が丘5名/日大前2名の集団参加消失）への意図的ガードで、単純化して上書き反映に変えてはならない**
    （`reflectParentOwnedSubmissionFields`・memory group-participation-save-paths）。

### D-2. オプション欄（開発用教室限定・フィーチャーフラグ）（監査 B2/C3・オーナー確定 2026-07-04）

- 提出ページに**オプション欄**（`optionChecks`・文言が入った行のみ表示）がある。文言は学年共通の `scheduleNotes`
  （`student:student-option-grade-{学年}-{行0..4}`＝`optionLabels`）から解決。
- ★**フィーチャーフラグ `studentScheduleOptionField` で「開発用教室のみ有効」**な機能（`isFeatureEnabledForClassroom`）。
  本番教室では出ない実験機能として正本化する。
- ★**学年解決の基準日が非対称**（回帰の温床・要注意）: オプションの学年は**今日基準**で解決するが、**提出本体（希望科目数・
  集団判定）は講習開始日基準**。この非対称を放置すると「学年が食い違いオプションが空文字で出る」不具合が再発しうる
  （`App.tsx` に回帰コメントあり）。基準日を変える際はこの非対称を必ず踏まえること。
- `optionChecks` は保護者所有フィールドとして D-1 と同じ **union 反映**（削減しない）で守られる。

## E. 提出・ロック・再提出 ★一段階モデルに再定義（監査 A2/C2・オーナー確定 2026-07-04）

- 「提出する」→確認→**提出完了画面**。提出後は本人から変更不可。
- 既に提出済みのリンクを開く／再送信すると、**提出者に「すでに提出済みです」画面を表示**（二重提出防止）。
- ★**提出＝即ロック＝即登録（一段階モデル）**:
  - 提出（QR の POST）は**即 `status='submitted'`** にし、以後の POST を **409 で拒否**（本人は提出した時点で再提出不可）。
  - 同時にサーバのスナップショット統合で **`countSubmitted: true`（＝室長の登録）** を立て、講師は自動配置・生徒は
    希望数/配置を反映する。**「提出」と「室長の登録（確定）」は独立ステップではなく一括で起こる。**
  - **再提出可能化は、室長が日程表側の登録を解除（`resetLectureSubmissionDoc`）したときのみ。**
  - **判断根拠**: 現行の「提出で即確定・戻すのは室長の登録解除のみ」は**二重提出防止と一貫しており安全側**。
    かつて正本には「提出後も室長が登録〈確定〉するまでは再提出可」という二段階の記述があったが、実装（POST 即
    `status='submitted'`・以後 409）と整合させ、この二段階の記述は**削除**した（実装を正とする・監査 A2）。
- 無効なリンクは「このリンクは無効です。**管理者にお問い合わせください**」と表示。

### E-1. 反映の永続化非対称と起動時 reconcile（監査 B3・オーナー確定 2026-07-04）

- QR 提出の反映は **capture（提出時）＋起動時 reconcile（あるべき状態への収束）の二段**で守る。
- **`countSubmitted`（specialSessions state）は workspace 自動同期で永続**するが、**盤面配置は手動保存（Cloud Function
  snapshot）でのみ永続**。この非対称ゆえ、手動保存前リロードで配置だけ揮発し `countSubmitted=true` が残るとスタックしうる。
- 起動時 `reconcileSubmittedTeacherPlacements`（純関数・boardMountKey で1回）が「`countSubmitted=true` なのに未配置の講師」を
  **冪等に再配置**して埋める。**過去の本番スタック（v1.5.357・提出したのに名前が出ない）への意図的ガードで消してはならない。**
- ただし**講師配置のみ自己修復があり、生徒の講習ストック消費デルタには自己修復が無い**（領域5 と連動）。

### E-2. 登録解除（リセット）のクリア粒度（監査 B9・オーナー確定 2026-07-04）

- 登録解除（`resetLectureSubmissionDoc`）でクリアするのは**提出内容6フィールド**（`unavailableSlots` / `subjectSlots` /
  `subjectDurations` / `groupClassParticipation` / `optionChecks` / `regularOnly`）と**提出時刻（`submittedAt`）のみ**。
  status は `pending` に戻す。
- **配布情報は維持**する: 本人名・期間・盤面配置（occupiedSlots）・休日（holidayDates）・限数（slotNumbers）・
  集団科目の選択肢（availableGroupClassSubjects）・オプション文言（optionLabels）。→ **同じ QR で再提出できる。**

### E-3. 登録削除の実装方式（監査 B5・オーナー確定 2026-07-04）

- 登録削除は **doc を残して pending に戻す（`resetLectureSubmissionDoc`）方式**で、doc の完全削除は行わない。
- `deleteLectureSubmissionDoc`（doc 完全削除）は**呼び出し 0 件のデッドコード**（TODO2 で「登録削除＝リセット」に統一した
  際の残骸・将来撤去候補）。

## F. 授業時間 90/60/45（⑤からの追加）

- **生徒が提出ページで授業時間 90／60／45分を選べる**ようにする。
- **デフォルト＝90分／未入力＝90分扱い。**
- 選択単位は**科目ごと**（希望科目数と同じ粒度）。※実装時に最終確認。

## G. 講習データの Excel 往復（監査 B7・オーナー確定 2026-07-04）

- 講習データ画面から **Excel 書き出し/取り込み**ができる（**特別講習／生徒日程入力／講師日程入力／説明**の 4シート）。
  照合は**講習ID優先→講習名**（`resolveSessionIdentity`）。生徒日程入力シートは参加不可コマ・希望科目数・通常のみ・提出済フラグを往復。
- ★**非対称（要注意）**: **授業時間（`subjectDurations`）・集団参加（`groupClassParticipation`）・オプション（`optionChecks`）は
  Excel 往復の対象外**（QR/日程表でのみ入出力）。Excel だけ再取込すると授業時間/集団/オプションが落ちるため、室長の体感に直結する。
