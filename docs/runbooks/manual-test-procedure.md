# Runbook: 実機確認テスト手順書（人間による最終受け入れ）

修正・新機能を入れたあと、**ユニット／ルールの自動テストが緑になったうえで**、本番反映前に
人間が実機で行う最終確認の手順書。自動テスト（純粋関数）が守れない「実環境の通し動作・画面表示・
保存往復・教室分離」を人の目で確認することが目的。

- 自動テストの方針 → [`docs/test-strategy.md`](../test-strategy.md)
- リリース判断の正本 → [safe-release スキル](../../.claude/skills/safe-release/SKILL.md)
- リリース前チェック票 → [`release-checklist.md`](./release-checklist.md)（本書はその「3. staging 実機確認」を詳細化したもの）

---

## 0. 大原則（厳守）

- **実機確認は staging（`https://komahyouapp-staging.web.app`）で行う。** 本番3教室
  （日大前／緑が丘／薬円台）は**読み取り専用**。検証で書き込みが要るデータは staging に
  ダミーを手で作る（本番データを staging へ持ち込まない＝個人情報混入の防止）。
- staging が未整備のときはローカル `npm run dev`（local モード）／エミュレータで代替し、
  **「実環境往復は未確認」である旨を記録に明示**する（[staging-environment スキル](../../.claude/skills/staging-environment/SKILL.md)）。
- デプロイ直後は**ハードリロード（Ctrl+Shift+R）**してから確認する（旧バンドルのキャッシュ事故防止）。

## 1. 事前準備

- [ ] 対象ブランチで `npm run lint` / `npm run test:unit` / `npm run build` が緑
- [ ] 保存・権限・教室分離を触ったなら `npm run test:rules` も緑（要 Java + エミュレータ）
- [ ] 「Deploy to Staging」ワークフローが緑（または代替環境を起動）
- [ ] staging を開き、**ハードリロード**した
- [ ] 確認に使う検証用データ（教室／生徒／講師／テンプレ）が staging に用意されている
- [ ] 画面右上などのバージョン表記が、今回デプロイした版になっている

## 2. 共通スモーク（**毎回・変更内容を問わず**通す）

どんな小さな変更でも、ここだけは通してから本番へ出す。1つでも×なら本番に出さない。

- [ ] **ログイン／教室切替**：ログインでき、担当する複数教室を切り替えられる。担当外教室が見えない。
- [ ] **コマ表が開く**：盤面が描画され、通常授業（`通)` ラベル）が曜日×時限に配置されている。
- [ ] **生徒の移動**：生徒を別の机／別コマへドラッグ移動できる。同コマ重複など不可な操作は弾かれ、
      盤面の状態が壊れない（移動前の席が消えるなど不整合が出ない）。
- [ ] **入れ替え（swap）**：埋まっている席同士で入れ替えができる。
- [ ] **休み／出席ステータス**：休みにすると薄字＋生徒日程表の休み欄に反映。出席トグルが効き、
      画面遷移しても保持される。
- [ ] **手動保存→再読込で保持**：保存ボタン（Cloud Function 経由）→ ページ再読込しても変更が残る。
- [ ] **生徒日程表**：通常授業が正しい回数でカウントされ、振替／講習が `振)` `講)` で区別表示される。
- [ ] **コンソールにエラーが出ていない**（DevTools の Console を一度確認）。

## 3. 変更箇所に応じた確認（該当ブロックだけ実施）

今回の修正・新機能が触れた領域のチェックリストを実施する。判断に迷ったら関連スペックを参照。

### A. コマ表・盤面操作（`spec-board-regular-placement.md`）
- [ ] 通常授業をテンプレどおり毎週配置／在籍期間外の週は出ない。
- [ ] **同日移動**は通常のまま（増コマは増コマのまま）で、移動済みヒントを残さない。
- [ ] **振替を元授業日へ戻すと通常扱い**に戻る。
- [ ] **休みマスへ別生徒を移動**しても、前の休み日付を引き継がない（既知回帰観点）。
- [ ] 講師を削除／生徒0でも盤面の講師名が保持される。
- [ ] 警告の赤表示：科目対応外／一コマ空け／講習の絶対制約／手動追加 がツールチップ付きで出る。
- [ ] **元に戻す／やり直し（undo/redo）**が期待どおり動く。

### B. 振替・講習ストック（`spec-makeup-stock.md` / `spec-lecture-stock.md`）
- [ ] 欠席登録で振替が発生し、コマは残り席は操作可。
- [ ] 振替の消化／ストック戻しで**残数が正しく増減**する。
- [ ] 講習の「削除＝希望−1」「ストックする＝戻す」の役割分担どおりに残数が動く。
- [ ] 授業時間 90／60／45 が反映される。
- [ ] ★**本番反映前は、残数が教室ごとにズレないことを開発用／staging で事前検証**（移行影響の確認）。

### C. 基本データ画面（`spec-basic-data.md`）
- [ ] 生徒・講師・テンプレの作成／編集／削除ができ、保存後に保持される。
- [ ] 在籍判定（入塾日〜退塾日＋高3卒業）が表示・配置に正しく効く。
- [ ] インポート（取り込み）のバリデーションが効く（不正データを弾く）。

### D. 特別講習・提出ページ／QR（`spec-special-session-submission.md`）
- [ ] 特別講習データの作成／編集ができる。
- [ ] **QR から提出 → 出席者一覧／盤面に反映**される（実 Functions 経由の経路）。

### E. 自動割振（`spec-auto-assign-rules.md`）
- [ ] 指定時限禁止・通常講師判定・ペア制約が守られる。
- [ ] 振替の同時割当が期待どおり。
- [ ] 割振後の盤面に制約違反の赤が残っていない。

### F. 日程表・PDF（`spec-schedule-pdf.md`）
- [ ] 「最新表示」ボタンで日程表が最新化される。
- [ ] PDF 出力のレイアウト・文字（エスケープ崩れが無いか）を目視。

### G. 保存・復元（`spec-save-restore.md`）
- [ ] 手動保存 → 再読込で席・ステータスが一致。
- [ ] JSON 復元で盤面が正しく戻る。
- [ ] **書き込み検証は開発用教室／staging のみ**（本番教室への復元・コピーは禁止）。

### H. 認証・教室権限・開発者画面（`spec-classroom-auth.md`）
- [ ] 権限のないユーザーが他教室／管理操作にアクセスできない。
- [ ] 管理者発行・メール変更などアプリ内操作が効く（staging で）。

## 4. 終了基準・記録

- [ ] **共通スモーク（§2）が全て○**、かつ**該当ブロック（§3）が全て○**。
- [ ] 新しい挙動・変更した挙動を守る**ユニットテストが同じコミットに入っている**
      （実機OKでもテスト無しではマージしない＝テストゲート厳守）。
- [ ] 確認結果を記録：いつ・どの版（version.json）・どの環境（staging/local）・確認した観点・
      残課題や「未確認の経路」を1行で残す。気になる挙動は GitHub Issue 化（`bug-triage` スキル）。
- [ ] ×が出たら本番へ出さず、原因を `dev-fix` で修正 → 自動テスト追加 → 再度本手順。

合格したら [`release-checklist.md`](./release-checklist.md) の「4. 記録」「5. リリース」へ進む。
