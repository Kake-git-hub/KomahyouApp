## 1. 保存

- 自動保存は**残す**。デバウンス **5秒**／連続操作中は最大 **20秒** でまとめて保存。
  - （現行 2秒／15秒 から変更）
- 保存先は **Firebase（クラウド）一本**。クラウドが唯一の正本。
  - ~~ブラウザ内（IndexedDB/localStorage）への業務データ二段階保存は**廃止**。~~
    **【2026-06-10 訂正】二段階保存は維持**（廃止しない）。クラウドが唯一の正本である点は不変だが、
    ローカル保存はクラッシュ復旧用キャッシュとして残す（パフォーマンス主因ではないため）。詳細は末尾「オーナー決定」。
  - 表示範囲などの UI 設定 localStorage は対象外（残す）。
- 手動「保存」ボタンは**常時表示・常に緑**。
  - 未保存あり → 押すと保存。保存済み → 「最新＝緑」のまま（押下は no-op）。グレーアウトは廃止。
- **閉じる／タブ切替で未保存があれば、離脱を止めて保存を促す。**
  - **【2026-07-04 確定・オーナー確認済み／C1：実装を正とする】** アプリ内の「手動保存を強制するダイアログ」は設けない。
    `beforeunload` ではブラウザ制約上アプリ独自のモーダルを表示できないため、**ブラウザ標準の離脱確認ダイアログ**
    （未保存・保存中・同期中のとき `event.preventDefault()`）で離脱を止め、あわせて**離脱時にローカル保存
    （同期 localStorage 書込）と Firebase 同期（`visibilitychange` hidden 時）を実行**することで代替する。
  - ~~日程表タブ以外~~ **タブ種別では分岐しない**。未保存があればタブに関係なく離脱確認を出す（実装が非依存で統一）。
- 自動保存（5秒）と離脱時の「離脱確認＋同期」は**両方残す**（二重の安全網）。

### 補足
- 自動保存の「重さ」は既に対策済み（盤面グリッド memo 化／週数の上限トリム／clone・stringify をデバウンス後へ移動）。
  自動保存自体は現状のパフォーマンス主因ではないため残す。

### 多端末競合の防止（楽観ロック）— 2026-07-04 確定・オーナー確認済み（監査 B1）

想定運用は「1端末でオンライン作業」だが、万一2端末目が同じ教室を編集した場合の**唯一の防波堤**として
楽観ロック（版数照合）を実装している。**過去のクロス汚染事故（他教室データでの上書き）への意図的な
破壊防止ガードであり、消してはならない。**

- クラウド保存は教室スナップショットの `version`（保存ごとに +1）で楽観ロックする。クライアントは
  「読み込んだ時点の版数（baseVersion）」を送り、サーバー版数と不一致なら**上書きせず拒否**する
  （`failed-precondition`。`functions/src/index.ts` `resolveOptimisticVersionDecision`）。
- 衝突時、このタブは (1) 内容を**JSON バックアップへ自動退避**、(2) 以後の**クラウド同期を停止**
  （古いベースでの再送ループを断つ）、(3) 「別の端末でこの教室のデータが更新されました。画面を再読み込みして
  ください」と**再読み込みを促す**。ローカル保存（IndexedDB）は継続するため編集内容はブラウザ内に残る。
- 版数は時刻比較では代替できない（多端末の時計ズレで誤判定するため必ず版数で照合する）。

## 2. 保存失敗時

- **原則、自動リトライしない。**
  - **【2026-07-04 確定・オーナー確認済み／C2：例外を明文化】** ただし**一時的（transient）な同期エラーに限り、
    最大3回まで自動再試行**する（`manualFirebaseSaveStability`。ネットワーク瞬断・409 等の一過性失敗を吸収するため）。
    **恒久失敗（非 transient）および版数衝突（多端末上書き）は再試行せず**、下記の JSON 退避＋作業指示へ進む
    （版数衝突は同じ古いベースで再送しても必ず失敗するため、クラウド同期を停止して再読み込みを促す。§1「多端末競合の防止」参照）。
- JSON バックアップを**自動でダウンロードフォルダへ書き出す**。
- ダイアログで「再ログインして、ダウンロードフォルダのバックアップを読み込んでください」と作業指示を出す。

### 保存時の3層防御（破壊防止ガード）— 2026-07-04 確定・オーナー確認済み（監査 B2）

Cloud Function `saveClassroomSnapshot` は保存時に次の3層で本番データを守る。**いずれも過去のクロス汚染・
偽 dirty 上書き事故（他教室データや空データでの上書き）への意図的な破壊防止ガードであり、消してはならない。**

1. **空データ上書き拒否**（`assertNoSnapshotDataLoss`）：既存に教室管理データがあるのに、空（管理データ0件）の
   ペイロードで上書きしようとした保存は `failed-precondition` で拒否する（前回件数を添えて中止）。
2. **保存後の読み戻し検証**：保存直後に Firestore から読み戻し、`payloadHash` が一致しなければ `internal` で失敗させる
   （サイレント破損＝壊れたまま「保存済み」表示になる事故を防ぐ）。
3. **saveId による冪等リプレイ**：同一 `saveId`・同一内容の再送は再保存せず検証済み結果を返す。同一 `saveId` で
   内容が異なる再送は `already-exists` で拒否する（ネットワーク再試行の二重書き込み・取り違えを防ぐ）。

## 4. 復元（4種類 → 1種類へ統合）

| 機能 | 扱い |
|------|------|
| 直前の Firebase 保存前へ戻す | **削除**（Undo で代替） |
| サーバーバックアップから復元 | 各教室画面から**削除**、**開発者画面のみ**へ移動 |
| ローカル自動バックアップから復元 | **削除** |
| JSON バックアップを読み込む | **残す（唯一の復元手段）** |

- 復元は「この教室だけ」に効き、他教室に影響しない（開発者画面のサーバー復元は教室選択モーダルで選んだ教室**だけ**を
  明示指定して保存し、他教室ドキュメントには一切書き込まない）。
- **復元時は確認モーダルで「元に戻せません」と警告してから実行。**
  - **【2026-07-04 確定・オーナー確認済み／C3】** この警告必須はそのまま維持する。監査時点では教室単位 JSON 読み込み・
    開発用教室への他教室読込には確認（上書き警告）があるが、**開発者画面のサーバーバックアップ復元モーダルには
    不可逆警告文言が無い**ため、実装側を本仕様に合わせて「復元すると選択教室の現在データは上書きされ、元に戻せません」を
    追加する（実装＋回帰テストは主セッションで対応）。

## 5. JSON バックアップ（書き出し／読み込み）

### 5-1. 教室単位（各教室画面）

- 書き出しは**開いている 1 教室分の“すべての状態・データ”**（テンプレート・設定・盤面・ストック等を含む）を 1 ファイルに。
- 読み込めば**完全復元**できること。読み込み時は確認（上書き警告）を出す。
- ファイル名は日本語＋日時入り。

### 5-2. 開発者画面：ワークスペース全体 — 2026-07-04 確定・オーナー確認済み（監査 B4）

- 教室単位（5-1）とは別に、**開発者画面のみ**「開発者バックアップ」として**ワークスペース全体
  （全教室・削除済み教室を含む）を1ファイルに書き出し／読み込み**できる。
- 読み込み時は**教室選択モーダル**で「復元する教室」を選ぶ（選んだ教室だけを復元し、選ばなかった教室は現状維持）。
  復元は選択教室のみを明示指定して保存し、他教室に影響しない（§4 と同じクロス汚染防止）。

### 5-3. 暗号化について — 2026-07-04 確定・オーナー確認済み（監査 B5）

- **JSON バックアップは平文（暗号化なし）**。読み込みにパスワードは不要。
- ローカル開発（Firebase 無効＝local モード）でのみ残っているパスワード入力プロンプトは**旧仕様の残骸**で、
  実際の復号処理は無い（将来撤去候補）。firebase モードでは渡さない（空文字）。

## 7. その他

- 危険操作後の「直前の状態に戻す」バナー（Undo）… 維持。
- **開発用教室へのデータ取り込み — 2026-07-04 確定・オーナー確認済み（監査 B6）**
  - かつての「他教室の現状データを再現（開発用教室へ**コピー**）」は**廃止**。in-memory の他教室参照を使う浅いコピーが

    **開発用教室（acting）だけ**へ書き込む（他教室には一切書き込まないため構造的に混入しない）。
  - 読み込み時は**取り違え防止の確認**を出す（読込元の規模＝生徒数／講師数／テンプレコマ数を提示し、想定と違えば中止を促す）。
  - 実行権限：**開発者アカウント、または開発用教室の室長**（開発用教室はサンドボックスのため任意教室を読み込める）。
    ~~室長へ一般開放~~ 一般教室の室長への開放は見送り（§末尾オーナー決定 2026-06-10）。

### QR 提出反映の永続化非対称と起動時の自己修復 — 2026-07-04 確定・オーナー確認済み（監査 B3）

QR 提出（講師/生徒）を盤面へ反映する機能には**永続化経路の非対称**があり、これを前提に自己修復を設けている。

- `countSubmitted`（提出反映フラグ。`specialSessions` 内）は**手動保存しなくても** workspace 自動同期で永続化される。
- 一方、QR 提出による**盤面への講師名配置（boardState の weeks/desks）は手動保存（Cloud Function snapshot）でしか
  永続化されない**。
- そのため手動保存前にリロードすると盤面配置だけ揮発し、`countSubmitted=true` は残るため取り込みでスキップされ
  「提出したのに出席可能コマに名前が出ない」状態でスタックしうる。
- **対処（維持）**：起動時（教室ロード/リロード毎に1回）に `reconcileSubmittedTeacherPlacements` が
  「`countSubmitted=true` なのに未配置の講師」だけを**冪等に**再配置する。既に一部でも配置済みの講師は触らない
  （手動調整・部分削除を尊重）。全コマ外したい場合は登録解除で対応する設計と整合。
