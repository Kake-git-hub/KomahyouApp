# 確定仕様：日程表コマ組み（生徒日程表からのD&Dコマ移動）— 正本

> 2026-07-08 オーナーQ&A（10問中4問＋共通2問）で確定。同日追加確定：
> **正式名称は「日程表コマ組み」**・自動割振ルール/警告はスコープ外・staging 先行リリース。
> 未実装（実装タスクは `docs/handoff-popup-sync-and-dnd.md` 参照）。
>
> 盤面の長押しD&D（`studentDragAndDropMove`・v1.5.338）の「日程表版」。
> **生徒日程表はコマ表（盤面）データの操作面になる**。コマ表の調整は生徒日程表を見て
> 生徒毎に調整することが多い、というオーナーの運用実態が動機。

## 0. ⚠️ 実装方式の転換（2026-07-08 追記・最重要）

**対話用日程表は React 化される（`spec-schedule-interactive-view.md`）。** 日程表ビューが
本体と同一 React ツリーになるため、この機能の実装は大幅に簡単になり、以下が**不要**になる：

- ポップアップ埋め込みJSでのD&D再実装（→ 通常の React DnD で実装）。
- 移動要求メッセージ（`schedule-student-move-request`）・postMessage 往復。
- 一過性リクエスト機構（`StudentScheduleRequest` の `mode:'move'` 拡張）・受信側の座標再解決。
- interaction-lock メッセージ・バージョンスキュー対策・`payloadAppVersion` 判定。

React 化後は、日程表ビューから**盤面の `executeMoveStudent` を直接呼ぶ**（同一ツリー・同一 state）。
机選択は React モーダル。移動結果は即座に同一 state 経由で日程表に反映される。

**したがって §C-3 後半・§D・§D-2 の「メッセージ／埋め込みJS／一過性リクエスト」に関する記述は
生成HTML方式を維持する場合の旧設計**として残す（§0 で置換）。一方、**データモデル上の確定事項は
方式に依存せず有効**：通常＝その回だけの振替（追加＋抑制の両方必須）／講習の rawKey 維持／
ルール・警告は無関係／手動保存で確定／source をエントリ id で特定／週範囲の自動拡張／Undo 対応。

## A. 背景と目的

- 現状、コマの移動は盤面（コマ表）上でしかできない。実際の調整作業は
  「生徒日程表で生徒の週間を見る → 盤面に切り替えて動かす → 日程表で確認」の往復になっている。
- 生徒日程表ポップアップ上で授業カードを長押しD&Dして空きコマへ動かせるようにし、
  生徒単位の調整を日程表から完結させる。

## B. 確定事項（オーナーQ&A 2026-07-08）

| # | 論点 | 決定 |
|---|------|------|
| 1 | 移動対象 | **通常授業・振替・講習・増コマ**（集団授業は対象外。集団は個別と独立の仕組みのため）。 |
| 2 | 通常授業の移動の意味 | **その回だけの振替**として扱う。既存の振替メカニズム（振替先の manual 追加＋振替元の抑制、**両方必須**）に乗せる。恒久的な曜日・時限変更（基本データ書き換え）ではない。 |
| 3 | 移動先の講師（机）の決定 | **ドロップ時にモーダルでその空きコマの机テーブルを表示し、コマ表のコマと同じ見た目で机を選ぶ**。自動割当ではなくユーザーが机を明示選択する。 |
| 4 | 制約違反時 | ~~盤面の手動操作と同じ（ソフト警告は警告表示しつつ配置許可）~~ → **2026-07-08 上書き：自動割振ルール・警告は日程表コマ組みでは一切無関係（評価も表示もしない）**。自動割当後に手動調整する前提の機能のため。判定するのは物理的な空き（満席・机なし・休校日）のみ。 |
| 5 | 保存 | 現行どおり**ローカル変更 → 手動保存ボタンで確定**。D&D移動だけの即サーバー保存はしない（保存経路の非対称を持ち込まない）。 |
| 6 | 展開 | ~~featureRollout（development-only → all-classrooms）~~ → **2026-07-08 上書き：まず staging だけに実装してオーナーチェックを待つ**。テストデータは日大前の現状データ（出席込み）のコピー（`tools/copy-prod-classroom-to-staging.mjs`）。本番への展開形はオーナーチェック合格後に決める（リアルタイム同期も同方針）。 |

## C. 挙動仕様

### C-1. ドラッグ開始（生徒日程表ポップアップ内）

- 生徒日程表の授業カード（`通)` / `振)` / `講)`）を**長押し（約250ms・盤面D&Dと同じ）**で掴む。
- 集団授業カード・出欠済みなど盤面で移動不可の状態は掴めない（盤面の可否判定に合わせる）。
- 掴んでいる間、ドロップ可能なセル（開校日かつその生徒の授業が入っていない空きセル）を
  視覚的に示す。休校日・期間外セルはドロップ不可。

### C-2. ドロップと机選択モーダル

- 空きセルにドロップすると、**そのコマ（日付×時限）の机テーブルをコマ表のコマと同じ見た目で
  表示するモーダル**がポップアップ内に開く：講師名・既存の着席生徒・空席が盤面と同じ配置で見える。
- ユーザーは空席（＝置ける席）を1つ選ぶ。満席の机・置けない席は選択不可の見た目にする。
- **モーダルが判定するのは物理的な空き（空席か否か）のみ**。自動割振ルール・警告
  （指定時限禁止など）は**評価も表示も一切しない**（2026-07-08 オーナー確定：
  日程表コマ組みは自動割当後の手動調整が前提のため、ルール・警告は無関係。
  ルール設定をポップアップへ送る必要もない＝機能1の軽量化方針とも整合）。
  盤面側に既存の警告表示があっても、この機能はそれに関与しない（変えない・消さない）。
- モーダルはキャンセル可能（キャンセルで移動不成立・何も変わらない）。

### C-3. 移動の実行（React 化後・§0 準拠）

- 席の確定で、日程表ビューから**盤面の移動ロジック `executeMoveStudent` / `computeStudentMove` を
  直接呼ぶ**（同一 React ツリーなのでメッセージ不要）。
  - **振替・講習**：盤面D&Dと同じ移動（入れ替え・各種ブロック判定を踏襲）。
    講習は**選択科目（rawKey）を維持**する（v1.5.364 の修正を巻き戻さない）。
  - **通常授業**：その回だけの振替として、移動先に振替を manual 追加し、移動元の当該日を
    抑制（suppression）する。**追加と抑制は必ず両方セットで行う**（7/20振替消失事故の教訓）。
- 移動が成立すれば同一 state 経由で日程表表示は自動更新（同期処理は不要）。
  移動が不成立（競合・満席・範囲外）なら盤面は変更せず、ビューに理由を表示する。
- 移動結果は未保存のローカル変更。手動保存ボタンで確定（盤面上の操作と同格）。

> （旧設計・生成HTML維持時）席確定でポップアップ→本体へ `schedule-student-move-request` を送り、
> `StudentScheduleRequest` の `mode:'move'` 拡張（Issue #46 パターン）で盤面へ渡して実行、
> 結果をポップアップへ返す。React 化により不要（§0）。

### C-4. 対象範囲・非対象

- 対象は**生徒日程表ポップアップのみ**（講師日程表からのD&Dは対象外・将来検討）。
- 同一セル内での順序入れ替え、授業が既にあるセルへの重ね掛けは初期リリースでは対象外
  （ドロップ先は空きセルのみ）。
- QR配布用など閲覧専用文脈での無効化：生徒日程表ポップアップは教室スタッフ操作面なので
  初期リリースでは常時有効でよい（featureRollout ゲートのみ）。

## D. 技術前提（React 化後・§0 準拠）

- 対話用日程表は本体と同一 React ツリーの React ビュー（`spec-schedule-interactive-view.md`）。
  D&Dは通常の React DnD（Pointer Events でマウス/タッチ両対応・盤面D&D `studentDragAndDropMove`
  の実装を踏襲）。埋め込みJS・`new Function` 構文検証・postMessage は対話には登場しない。
- 表示データ（各コマの机・講師名・着席生徒・授業種別）は盤面 state から算出する既存ロジック
  （`serializeCells` 由来の純関数）を共有して得る。机選択モーダルは React コンポーネント。
- 移動は盤面の `executeMoveStudent` / `computeStudentMove` を直接呼ぶ（同一ツリー・同一 state）。

### D-2. 方式に依存しない確定事項（React でもテストで固定する）

1. **週範囲の自動拡張**：日程表の表示期間は盤面が読み込んでいる週より広いことがある。
   移動先セルが `boardState.weeks` に無ければ `ensureWeeksCoverDateRange`（既存）で拡張してから
   移動。拡張できない日付（運用範囲外）は不成立として理由を返す。
2. **source 特定の頑健性**：移動元カードは `entryId`（`statusSlots` エントリ id）＋`studentId`＋
   日付＋時限で盤面エントリを特定する。同一生徒が同コマに2授業（2科目）を持つケースがあるため、
   **名前や科目だけで特定しない**。実行時点でエントリが無ければ不成立として理由を返す。
3. **Undo 対応**：D&D移動は盤面操作と同じ履歴に乗り、盤面の Undo で1操作として戻せる
   （`executeMoveStudent` 再利用で自然に満たされるはずだが、テストで固定）。
4. **教室整合**：移動は現在開いている盤面（`actingClassroomId`）に対してのみ作用する
   （React 同一ツリーなので別教室混入は構造上起きないが、テストで固定）。

> （旧設計・生成HTML維持時の追補）interaction-lock による二重操作防止、
> `classroomStorageKey` 付きメッセージの教室ガード、`payloadAppVersion` によるバージョンスキュー
> 自己修復は、いずれも postMessage 境界のための機構。React 化により不要（§0）。

## E. 受け入れ条件

1. 生徒日程表で `振)` カードを空きコマへD&D → 机選択 → 盤面の該当コマに移動しており、
   日程表表示も更新される。
2. `講)` カードの移動で選択科目が変わらない（rawKey 維持・v1.5.364 の回帰なし）。
3. `通)` カードの移動で「移動先に振)が追加」「移動元の当該日が抑制」の**両方**が起きる。
   移動元の他の週の通常授業は不変。
4. 満席のコマ・休校日にはドロップ/選択できない。自動割振ルールに反する席でも
   物理的に空いていれば警告なしで配置できる（2026-07-08 確定：ルール・警告は無関係）。
5. モーダルのキャンセルで盤面が一切変わらない。
6. 移動後に盤面を手動保存 → リロードで移動が永続化。保存せずリロードなら移動は消える
   （盤面操作と同格＝既存アーキの想定どおり）。
7. staging 環境判定が無効な文脈では日程表コマ組みが始まらない（本番3教室は影響ゼロ）。
8. 集団授業カードは掴めない。
9. 盤面が読み込んでいない先の週へ移動しても週が自動拡張されて成立する（D-2-1）。
10. source エントリが見つからない・target が埋まっている場合、盤面は不変で理由が表示される（D-2-2）。
11. D&D移動が盤面の Undo で1操作として戻せる（D-2-3）。
12. ドック表示・ポップアウト表示のどちらからでも同じD&D操作ができる（同一コンポーネント）。

## F. メリット・デメリット

**メリット**

- 生徒毎の調整が日程表上で完結し、盤面との往復がなくなる（オーナーの主要ワークフロー短縮）。
- 移動ロジックは盤面の既存実装（`executeMoveStudent`）を再利用するため、
  盤面とポップアップで挙動が食い違わない。
- 机選択モーダルにより「どの講師に付くか」をユーザーが常に把握・選択できる
  （自動割当のブラックボックス感がない）。

**デメリット・リスク**

- 対話用日程表の React 化（`spec-schedule-interactive-view.md`）が前提。その土台の実装コストが
  かかる（ただしこの機能自体は土台の上では埋め込みJS不要・メッセージ不要で簡潔になる）。
- 同一 React ツリーなので空席情報は常に最新だが、`executeMoveStudent` 実行時の
  ブロック判定（満席・範囲外）は必ず通す。不成立なら盤面不変で理由表示。
- 「通常＝その回だけの振替」の意味をユーザーが誤解する余地（恒久変更と思い込む）。
  モーダルまたはドロップ後の表示で「この回のみ振替」表記を添える。

## G. 本番環境への影響範囲

- **データモデル・Firestore・Cloud Functions への変更なし**。移動は既存の盤面ローカル状態の
  変更（既存の振替・移動と同じデータ形）で、保存経路も既存の手動保存のみ。
- 触るのはフロントのみ：新規の React 日程表ビュー（D&D・机選択モーダル）、
  `src/components/schedule-board/ScheduleBoardScreen.tsx`（`executeMoveStudent` の共有）、
  `src/utils/featureRollout.ts`（staging 環境判定）。生成HTML（`scheduleHtml.ts`）は
  印刷用に残置し、対話用途では使わない。
- **リリースは staging 先行（2026-07-08 確定）**：feature ブランチを staging にだけデプロイし、
  オーナーチェック合格まで main（＝本番）にマージしない。本番3教室への影響はチェック合格まで
  ゼロ。staging のテスト教室は開発用教室IDではないため、**staging 環境判定
  （firebaseConfig の projectId === 'komahyouapp-staging'）で機能を有効化する**。
  誤操作リスクは「盤面での手動移動と同等」（未保存のローカル変更なので保存前ならリロードで戻せる）。
- React 化により、旧方式の「埋め込みJS構文エラーで日程表全停止」という最悪ケースは消える。
  代わりに `spec-schedule-interactive-view.md` §E のメモリ規律を守ること（怠ると heap 増加）。
