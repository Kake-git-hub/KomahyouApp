# 確定仕様：生徒日程表からの長押しD&Dコマ移動 — 正本

> 2026-07-08 オーナーQ&A（10問中4問＋共通2問）で確定。未実装（実装タスクは
> `docs/handoff-popup-sync-and-dnd.md` 参照）。
>
> 盤面の長押しD&D（`studentDragAndDropMove`・v1.5.338）の「日程表ポップアップ版」。
> **生徒日程表はコマ表（盤面）データの操作面になる**。コマ表の調整は生徒日程表を見て
> 生徒毎に行うことが多い、というオーナーの運用実態が動機。

## A. 背景と目的

- 現状、コマの移動は盤面（コマ表）上でしかできない。実際の調整作業は
  「生徒日程表で生徒の週間を見る → 盤面に切り替えて動かす → 日程表で確認」の往復になっている。
- 生徒日程表ポップアップ上で授業カードを長押しD&Dして空きコマへ動かせるようにし、
  生徒単位の調整を日程表から完結させる。

## B. 確定事項（オーナーQ&A 2026-07-08）

| # | 論点 | 決定 |
|---|------|------|
| 1 | 移動対象 | **通常授業・振替・講習**（集団授業は対象外。集団は個別と独立の仕組みのため）。 |
| 2 | 通常授業の移動の意味 | **その回だけの振替**として扱う。既存の振替メカニズム（振替先の manual 追加＋振替元の抑制、**両方必須**）に乗せる。恒久的な曜日・時限変更（基本データ書き換え）ではない。 |
| 3 | 移動先の講師（机）の決定 | **ドロップ時にモーダルでその空きコマの机テーブルを表示し、コマ表のコマと同じ見た目で机を選ぶ**。自動割当ではなくユーザーが机を明示選択する。 |
| 4 | 制約違反時 | **盤面の手動操作と同じ**。物理的に置けない（満席・机なし等）は選択不可、ソフト警告（指定時限禁止等）は警告表示しつつ配置は許可。盤面と日程表で「置ける/置けない」の非一貫を作らない。 |
| 5 | 保存 | 現行どおり**ローカル変更 → 手動保存ボタンで確定**。D&D移動だけの即サーバー保存はしない（保存経路の非対称を持ち込まない）。 |
| 6 | 展開 | **featureRollout（`development-only` → 実機検証後 `all-classrooms`）**。 |

## C. 挙動仕様

### C-1. ドラッグ開始（生徒日程表ポップアップ内）

- 生徒日程表の授業カード（`通)` / `振)` / `講)`）を**長押し（約250ms・盤面D&Dと同じ）**で掴む。
- 集団授業カード・出欠済みなど盤面で移動不可の状態は掴めない（盤面の可否判定に合わせる）。
- 掴んでいる間、ドロップ可能なセル（開校日かつその生徒の授業が入っていない空きセル）を
  視覚的に示す。休校日・期間外セルはドロップ不可。

### C-2. ドロップと机選択モーダル

- 空きセルにドロップすると、**そのコマ（日付×時限）の机テーブルをコマ表のコマと同じ見た目で
  表示するモーダル**がポップアップ内に開く：講師名・既存の着席生徒・空席が盤面と同じ配置で見える。
- ユーザーは空席（＝置ける席）を1つ選ぶ。満席の机・置けない席は選択不可の見た目にする。
- **モーダルが判定するのは物理的な空き（空席か否か）のみ**。ソフト警告（指定時限禁止など）は
  モーダルでは判定せず、**移動成立後に盤面・日程表の赤字警告として表示**する
  （2026-07-08 精緻化：自動割振ルール設定はポップアップのペイロードに含まれておらず、
  含めると機能1の軽量化方針と矛盾する。盤面の手動配置も「置いた後に赤字警告」なので、
  この形が「盤面の手動操作と同じ」の確定事項に合致する）。
- モーダルはキャンセル可能（キャンセルで移動不成立・何も変わらない）。

### C-3. 移動の実行（本体側）

- 席の確定でポップアップ → 本体へ移動要求メッセージを送る。本体（盤面）は既存の移動ロジック
  （`executeMoveStudent` / `computeStudentMove`）を再利用して盤面状態を更新する。
  - **振替・講習**：盤面D&Dと同じ移動（入れ替え・各種ブロック判定を踏襲）。
    講習は**選択科目（rawKey）を維持**する（v1.5.364 の修正を巻き戻さない）。
  - **通常授業**：その回だけの振替として、移動先に振替を manual 追加し、移動元の当該日を
    抑制（suppression）する。**追加と抑制は必ず両方セットで行う**（7/20振替消失事故の教訓）。
- 移動が成立したら該当生徒のポップアップ表示を即時更新（force同期1回）。
  移動が不成立（競合・ブロック）ならポップアップに理由を表示し、盤面は変更しない。
- 移動結果は未保存のローカル変更。手動保存ボタンで確定（盤面上の操作と同格）。

### C-4. 対象範囲・非対象

- 対象は**生徒日程表ポップアップのみ**（講師日程表からのD&Dは対象外・将来検討）。
- 同一セル内での順序入れ替え、授業が既にあるセルへの重ね掛けは初期リリースでは対象外
  （ドロップ先は空きセルのみ）。
- QR配布用など閲覧専用文脈での無効化：生徒日程表ポップアップは教室スタッフ操作面なので
  初期リリースでは常時有効でよい（featureRollout ゲートのみ）。

## D. 技術前提（実装調査済みの事実）

- 生徒日程表ポップアップは `window.open` した別ウィンドウに `src/utils/scheduleHtml.ts` が
  生成した HTML＋**テンプレートリテラル内の埋め込みJS**を流し込む構造。
  ⚠️ エスケープ崩れがビルドをすり抜けて実行時全停止する既知の罠あり
  （memory `komahyou-schedulehtml-embedded-script`・`new Function` 構文検証テストが存在）。
- ポップアップのペイロード（`DATA.cells`）には**各コマの机（`desks`）・講師名・着席生徒・
  授業種別が含まれている**（`serializeCells`）。机選択モーダルはポップアップ側だけで描画できる。
- ポップアップ → 本体の書き込みメッセージは前例多数（`schedule-student-count-save` /
  `schedule-student-unavailable-save` 等）。本体側ハンドラは `App.tsx` の
  `handleScheduleRangeMessage` 周辺。
- App → 盤面への一過性リクエスト機構が既にある：`StudentScheduleRequest`
  （`requestId` + `mode: 'unassign'`）＋ `consumeStudentScheduleRequest`（処理後 null 化）。
  **Issue #46（再マウント再発火）対策済みのこのパターンを必ず踏襲・拡張する**
  （`mode: 'move'` の追加）。独自の新経路を作らない。

## D-2. 詳細追補（2026-07-08 ギャップ精査で確定）

1. **盤面の週範囲外への移動**：ポップアップの表示期間は盤面が現在読み込んでいる週より
   広いことがある。移動先セルが `boardState.weeks` に無い場合は
   `ensureWeeksCoverDateRange`（既存）で対象日まで盤面を拡張してから移動を実行する。
   拡張できない日付（教室の運用範囲外）は不成立として理由を返す。
2. **二重操作防止**：移動要求を送ってから結果メッセージを受け取るまで、ポップアップは
   既存の interaction-lock 機構（`schedule-shared:interaction-lock` /
   `schedule-force-release-interaction`）を使って追加のD&D・編集操作をロックする。
   結果受信またはタイムアウト（数秒）で必ず解除する（ロックしっぱなし事故を作らない）。
3. **教室ガード**：`schedule-student-move-request` に `classroomStorageKey` を含め、
   本体側で `actingClassroomId` と一致しないメッセージは黙って破棄する
   （古い別教室ポップアップからの要求で現教室の盤面を書き換えない）。
4. **Undo 対応**：D&D移動は盤面操作と同じ履歴に乗り、盤面の Undo で1操作として戻せること
   （`executeMoveStudent` 再利用により自然に満たされるはずだが、テストで固定する）。
5. **source 特定の頑健性**：ポップアップのカードは `entryId`（`statusSlots` エントリ id）＋
   `studentId`＋日付＋時限で盤面上のエントリを特定する。同一生徒が同コマに2授業
   （2科目受講）持つケースがあるため、**名前や科目だけで特定しない**。実行時点の盤面で
   エントリが見つからない（同期ずれで既に移動/削除済み）場合は不成立として理由を返す。
6. **バージョンスキュー対策**：機能1の D-2-2 と同じ仕組みに乗る（ペイロードの
   `payloadAppVersion` 不一致でポップアップ再オープン）。移動要求メッセージも
   本体側で未知/不正形を黙って無視する防御的ハンドリングとし、旧版ポップアップからの
   要求で例外を出さない。

## E. 受け入れ条件

1. 生徒日程表で `振)` カードを空きコマへD&D → 机選択 → 盤面の該当コマに移動しており、
   日程表表示も更新される。
2. `講)` カードの移動で選択科目が変わらない（rawKey 維持・v1.5.364 の回帰なし）。
3. `通)` カードの移動で「移動先に振)が追加」「移動元の当該日が抑制」の**両方**が起きる。
   移動元の他の週の通常授業は不変。
4. 満席のコマ・休校日にはドロップ/選択できない。ソフト警告席は警告表示のうえ配置できる
   （盤面手動操作との一貫）。
5. モーダルのキャンセルで盤面が一切変わらない。
6. 移動後に盤面を手動保存 → リロードで移動が永続化。保存せずリロードなら移動は消える
   （盤面操作と同格＝既存アーキの想定どおり）。
7. 盤面キー再マウント（教室切替等）で移動リクエストが再発火しない（Issue #46 型の回帰なし）。
8. featureRollout 無効教室では長押ししてもD&Dが始まらない。
9. 集団授業カードは掴めない。
10. 盤面が読み込んでいない先の週へ移動しても週が自動拡張されて成立する（D-2-1）。
11. 移動要求中は追加操作がロックされ、結果受信/タイムアウトで必ず解除される（D-2-2）。
12. 盤面側で source エントリが見つからない・target が埋まっている場合、盤面は不変で
    ポップアップに理由が表示される（D-2-5・古いスナップショットからの要求）。
13. D&D移動が盤面の Undo で1操作として戻せる（D-2-4）。

## F. メリット・デメリット

**メリット**

- 生徒毎の調整が日程表上で完結し、盤面との往復がなくなる（オーナーの主要ワークフロー短縮）。
- 移動ロジックは盤面の既存実装（`executeMoveStudent`）を再利用するため、
  盤面とポップアップで挙動が食い違わない。
- 机選択モーダルにより「どの講師に付くか」をユーザーが常に把握・選択できる
  （自動割当のブラックボックス感がない）。

**デメリット・リスク**

- ポップアップ埋め込みJSの大幅追加（長押しD&D＋モーダル描画）。テンプレートリテラル内文字列
  なのでエスケープ事故・実行時全停止のリスクが構造的にある（構文検証テストで防ぐ）。
- ポップアップの `DATA.cells` は同期時点のスナップショット。盤面がその後変わっていると
  モーダルの空席情報が古い可能性 → 本体側で実行時に再検証し、不成立なら理由を返す設計で吸収
  （リアルタイム同期機能の併用でズレ自体も縮む）。
- 「通常＝その回だけの振替」の意味をユーザーが誤解する余地（恒久変更と思い込む）。
  モーダルまたはドロップ後の表示で「この回のみ振替」表記を添える。

## G. 本番環境への影響範囲

- **データモデル・Firestore・Cloud Functions への変更なし**。移動は既存の盤面ローカル状態の
  変更（既存の振替・移動と同じデータ形）で、保存経路も既存の手動保存のみ。
- 触るのはフロントのみ：`src/utils/scheduleHtml.ts`（D&D・モーダル・移動要求メッセージ）、
  `src/App.tsx`（メッセージ受信 → 一過性リクエスト発行）、
  `src/components/schedule-board/ScheduleBoardScreen.tsx`（リクエスト処理・移動実行）、
  `src/utils/featureRollout.ts`（キー追加）。
- featureRollout により本番3教室は昇格コミットまで挙動不変。誤操作リスクは
  「盤面での手動移動と同等」（未保存のローカル変更なので保存前ならリロードで戻せる）。
- 最悪ケース（埋め込みJSの構文エラー）は**日程表ポップアップ全体の実行時停止**。
  `new Function` 構文検証テストと staging 実機確認を必ず通す。
