# 開発ルール

## 目的

コマ表、振替ストック、日程表、講習期間表示の変更で回帰を残さないため、実装修正後に定期的な確認を必ず実施する。

## 実施タイミング

- コマ表の配置ロジック、振替ロジック、休日処理を触った直後
- 生徒日程表 / 講師日程表 / PDF / 講習期間帯を触った直後
- リリース相当の区切り前
- 少なくとも週 1 回の保守確認時
- Playwright はユーザーから明示指示があったときだけ実行する

## 必須チェック

1. `npm run build`
2. 振替ストック計算または PDF 出力を触った場合は `npm run test:unit`
3. `npx playwright test tests/schedule-board.spec.ts` は E2E 指示時のみ

## ゴールデンスナップショット（回帰の自動検知）

固定の代表教室データを純関数に通し、結果を読みやすい digest 文字列でスナップショット固定する。
`npm run test:unit` で毎回走り、**意図せずスナップショットが変わったら回帰の可能性が高い。**
差分を読んで原因を特定する。仕様を意図的に変えたときだけ各テストを `npx vitest -u <path>` で
更新し、差分が仕様どおりか必ず目視確認する。

### コマ配置 `buildScheduleCellsForRange`

- 入力 fixture: `src/components/schedule-board/__fixtures__/sampleClassroom.ts`
  （フル稼働 / 月途中開始 / 月途中退塾 / 休日 / ペア授業 / 講師のみデスクを意図的に含む）
- テスト: `boardSnapshot.test.ts` / スナップショット: `__snapshots__/boardSnapshot.test.ts.snap`
  （`日付 (曜) 限 | 講師 | 生徒(学年/科目)` の 1 行 = 1 配置）

### 振替ストック `buildMakeupStockEntries`

- 入力 fixture: `src/components/schedule-board/__fixtures__/sampleMakeupStock.ts`
  （未消化ストック / 手動ストックの部分消化＝再ストック / 過剰配置のマイナス残 を含む）
- テスト: `makeupStockSnapshot.test.ts` / スナップショット: `__snapshots__/makeupStockSnapshot.test.ts.snap`
  （`key | 残 | 自動/手動 | 配置振替 | 通常/上限/超過 | 次源 | 理由` の 1 行 = 1 ストック行）

### 日程表 想定回数 `buildExpectedRegularOccurrences`

- 入力 fixture: `src/components/schedule-board/__fixtures__/sampleScheduleOccurrences.ts`
  （フル稼働 / 月途中開始 / 月途中退塾 を含む。row 期間を月内へ束ねている）
- テスト: `src/utils/scheduleOccurrencesSnapshot.test.ts`
  / スナップショット: `src/utils/__snapshots__/scheduleOccurrencesSnapshot.test.ts.snap`
  （`生徒__科目 | 計N | 日付…` の 1 行 = 1 生徒科目）
- 注意: この関数は休日では間引かない「想定回数」ベースライン。表示レンジではなく
  row の授業期間（空なら schoolYear 全体）で返すため、盤面用 fixture とは別物を使う。

### 講習ストック `buildLectureStockEntries`

- 入力 fixture: `src/components/schedule-board/__fixtures__/sampleLectureStock.ts`
  （提出済み複数科目 / 未提出除外 / 通常のみ除外 / 0回除外 / 複数セッション整列）
- テスト: `lectureStockSnapshot.test.ts` / スナップショット: `__snapshots__/lectureStockSnapshot.test.ts.snap`
  （`セッション | 生徒 | 科目 | 希望回数` の 1 行 = 1 ストック行）

### 観点を増やすとき

fixture は「実際の教室データの縮図」。回帰観点を増やしたいときは fixture に生徒/授業/振替を足し、
必要なら各 `*Snapshot.test.ts` 後半の不変条件アサートも追加する。
`today` 依存の関数は必ず固定 `today` を渡し、決定的に保つ。

## Playwright で最低限守る観点

- 基本データの追加、編集、削除がコマ表へ反映されること
- 通常授業の契約回数は `各月一律 4 回` ではなく、月途中開始 / 終了時は有効な残り週数分だけになること
- 5 週ある月だけ 5 回目以降の生徒配置を止め、講師だけ残すこと
- 振替ストックの計算、消化、再ストックが崩れないこと
- `src/components/schedule-board/makeupStock.test.ts` の単体ケースで、休日不足 / 手動追加除外 / 空きコマ不足が崩れないこと
- 振替中 / 移動中に同コマ重複を拒否し、選択状態を維持すること
- 日程表タブが別タブ表示、日付入力即反映、講習期間セレクター反映、ライブ追従更新を維持すること
- 通常授業を stock へ回した未割当分が日程表に残らないこと
- 通常回数 / 講習回数の警告スタンプがそれぞれの表の直下に出ること
- 生徒日程の `算` / `数` が学年に応じて片方だけ表示されること
- 講習期間帯と休日グレーが表示されること
- PDF の時間列が中央配置の大きい文字で出力され、stock パネルが混入しないこと

## ドキュメント更新ルール

- 仕様や運用ルールを変えたら `README.md` と `.github/copilot-instructions.md` を同じ変更内で更新する
- 日程表の反映ロジックや stock 表示ルールを変えたら `docs/schedule-popup-design.md` も更新する
- PDF 出力のレイアウトや文字組みを変えたら `docs/board-pdf-design.md` も更新する
- テスト運用や確認手順を変えたらこのファイルも更新する

## 失敗時の扱い

- build が落ちた状態で作業完了扱いにしない
- Playwright を実行した場合は、その失敗も解消するか既知課題として記録する
- 既知の失敗として残す場合は、失敗テスト名、原因、暫定回避、次の対応を記録する
