# 回帰防止ガード（過去の修正を巻き戻さない）

このアプリは**過去の修正を積み上げる**前提。後の変更で直したバグを戻すと信頼を失う。
オーナーの厳命であり最優先級のルール（CLAUDE.md「回帰防止ルール」）。

## 書き換える前に必ずやること
1. その箇所が**過去に何を直したか**を確認する:
   - `git log -L <開始>,<終了>:<ファイル>` … 関数単位の変遷
   - `git log -S "<該当コード片>"` … その文字列が入った/消えたコミット
   - `git blame <ファイル>` … 行ごとの由来
2. 近くの「**回帰防止**」「**fix**」「**過去コミットID**」コメントは**意図的なガード**。
   消さない・薄めない・短くしない。

## 典型的な踏み抜きパターン
- **spread 単純化の罠**: `{ ...a, ...b }` は、過去に明示していたフィールド補完
  （例: `managedStudentId: b.x ?? a.x`）を消すと回帰する。短く見えても挙動が変わる。
  - 実例: `mergeManagedDeskLesson` の生徒同一性補完（commit `6793374`）が後のマージ（`2dce7b4`）で
    `{ ...managedStudent, ...student }` に巻き戻り、生徒日程表で通常授業がカウントされない不具合が再発
    （2026-06-10 に再修正＋テスト追加）。
- **マージ衝突の解決**: 一方の修正を捨てて他方で上書きしない。**両側の修正を必ず両立**させる。
- **埋め込みスクリプトのエスケープ**: `scheduleHtml.ts` はテンプレートリテラル内の文字列。
  正規表現の二重エスケープ等を崩すとビルドをすり抜けて実行時に全停止する（memory: schedulehtml-embedded-script）。

## 回帰テストの付け方（必須）
- バグを直したら**必ず回帰防止テストを追加**する。
- **修正なしで落ち・修正ありで通る**ことを確認（テストが本当にそのバグを捕まえるか）。
- **ゴールデンスナップショットだけに頼らない**。リライトで保護が消える。純粋関数に切り出して
  入力→出力を固定するのが堅い（例: `shouldWarnForbiddenPeriod` の切り出し）。

## 仕上げ
- コミット前に `git diff` を通読し、意図しない巻き戻し・コメント削除が無いか確認。
- 不安な箇所は `regression-reviewer` エージェントにレビューさせる。
