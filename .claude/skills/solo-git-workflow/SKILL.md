# 一人開発向け Git ワークフロー + 回帰防止

このスキルは、一人で開発しているリポジトリでコードを編集する際の進め方を定める。
2つの目的がある:

1. **`main` を常に動く安全な状態に保つ**(Git運用)
2. **修正したら別の場所が壊れる「回帰(デグレ)」を防ぐ**(変更前後の確認)

一人開発なのでレビューや承認プロセスは省き、判断はすべて自分(Claude)で行ってよい。
ただし「壊れたら困る範囲かどうか」と「既存の動作を壊していないか」だけは毎回見極める。

## 大原則

- **`main` を壊さない。** `main` はいつでも動く状態を保つ。リスクのある変更を `main` で直接やらない。
- **触る前に現状を理解する。** 何が今正しく動いているかを把握してから変更する。これが回帰防止の出発点。
- **変更は最小限・局所的に。** 頼まれていない範囲まで広げない。回帰の最大の原因は「ついで」の変更。
- **変更後は必ず確認する。** 直した箇所だけでなく、影響を受けそうな既存機能も動くか確かめる。
- **こまめにコミットする。** 動く状態を細かく記録し、回帰したらすぐ戻れるようにする。
- **Push は作業の切れ目で。** 毎コミットごとに Push しなくてよい。区切りがついたら Push する。

## 編集前チェックと更新リスト(回帰防止・厳守)

このリポジトリは **CI が main への push ごとに patch バージョンを自動 bump** し、
`chore(release): vX.Y.Z デプロイに伴うバージョン更新` を main に戻す
(`.github/workflows/deploy-firebase-hosting.yml` → `tools/bump-version.mjs`)。
このため **ローカルの main はしばしばデプロイ済み(= origin/main / ライブ)より遅れている**。
古いベースのまま編集すると、CI が直近で入れた修正やリリースコミットを巻き戻し、
過去に直したバグを再発させる(CLAUDE.md の回帰事例と同根)。これを防ぐため、
**編集着手前**と**編集ごと**に以下を必ず行う。

### A. 編集を始める前:デプロイ済み最新との同期確認(必須)

コードを1行でも触る前に:

1. リモートを取得して、ローカル main が origin/main に追いついているか確認する。
   ```bash
   git fetch origin
   git status -sb              # "behind" と出たらローカルが遅れている
   ```
   - 遅れていれば `git switch main && git pull --ff-only` で最新化してから着手する。
   - 未コミットの変更があって pull できないときは、先に退避(commit / stash)する。
2. デプロイ済み(ライブ)のバージョンと、ローカルの `package.json` の version を突き合わせる。
   ```bash
   node -p "require('./package.json').version"                # ローカル
   curl -s https://komahyouapp-prod.web.app/version.json      # ライブ(デプロイ済み最新)
   ```
   - 一致していれば「ローカル = デプロイ済み最新」。安心して着手してよい。
   - **ライブの方が新しいまま編集を始めない。** 古ければ手順1で最新化する。
3. 着手前のベースラインを固める(下記「1. 変更前:現状を把握する」に従い、`git status` がクリーンであること)。

> ⚠️ バージョンは **CI が自動で上げる**。**Claude は `package.json` の version を手動で書き換えない**
> (二重 bump・版ズレの原因)。Claude がやるのは「最新確認」と「更新リストの記載」だけ。

### B. 編集ごと:更新リスト `CHANGELOG.md` の記載(必須)

コードを編集したら、**そのコミットと同じまとまりで必ず**ルートの更新リスト `CHANGELOG.md` に追記する。
「どの版で何を変えたか」を残し、回帰したとき原因の版・コミットを素早く特定するための回帰防止記録。

1. `CHANGELOG.md` 先頭の `## 未リリース` セクションに、変更内容を簡潔に1行で追記する。
   - 書式: `- <種別>: <何を・なぜ>`(種別 = `feat` / `fix` / `refactor` / `style` / `docs` / `chore`)
   - 例: `- fix: 生徒日程表で通常授業がカウントされない回帰を修正(mergeManagedDeskLesson の生徒同一性補完を復元 / 6793374)`
   - 触ったファイル名や関連コミットIDを括弧で添えると後追いが速い。
2. main にマージ(= デプロイ)する直前に、`## 未リリース` の見出しを **次にデプロイされる版**でラベル付けする。
   - 次の版 = 現在の `package.json` の version の patch **+1**(CI が deploy 時に +1 するため)。
   - 例: `package.json` が `1.5.329` なら → 見出しを `## v1.5.330 (YYYY-MM-DD)` に書き換え、
     その上に新しい空の `## 未リリース` を作る。
3. `git diff` で確認後、**コードと `CHANGELOG.md` を一緒にコミットする**。

これで「編集ごとに版が上がり(CI)、その版に対応する更新リストが残る(Claude)」が両立する。

### C. コミット〜main マージは自分の判断で進める(常時許可・オーナー指示)

オーナーの方針により、**毎回「コミットして」「マージして」と言われなくても、Claude は自分の判断で
コミットから main へのマージ(= デプロイ)まで進めてよい**(2026-06-27 指示)。具体的には:

- 編集が一区切りついたら、コードと `CHANGELOG.md` を一緒にコミットする。
- `## 未リリース` を次の版(`package.json` の patch +1)でラベル付けし、新しい空の `## 未リリース` を作る。
- ブランチで作業した場合は main にマージする。main への push で CI が自動 bump・自動デプロイする。
- これら一連を**確認を求めずに実行してよい**。完了後に「何を・どの版で・どこへ反映したか」を簡潔に報告する。

ただし次は**この常時許可の対象外**で、従来どおり慎重に扱う(勝手に進めない):

- **本番データへの書き込みを伴う操作**(CLAUDE.md「本番データ保護ルール」最優先)。書き込みは開発用教室のみ。
- **回帰確認を飛ばすこと**。マージ前に必ず下記「回帰防止の手順」(build / 関連テスト / `git diff`)を通す。
  確認できない動作はマージ前にオーナーへ具体的に伝える。
- **履歴の破壊的操作**(`reset --hard`、`push --force`、共有済みブランチの rebase)。これは明示確認が必要。
- **壊れたら困る大きな変更を main で直接始めること**。リスクがあるものは従来どおりブランチを切る。

## 回帰(デグレ)防止の手順

修正やリファクタリングで「直したはずなのに別の場所が壊れる」のを防ぐため、以下を必ず守る。

### 1. 変更前:現状を把握する(土台を固める)

- **まず動く状態をコミットしておく。** 作業前にコミットされていない変更がないか `git status` で確認し、
  あれば一旦コミットするか退避する。これで「直前の動く状態」にいつでも戻れる。
- **変更対象の周辺を読む。** これから触る関数・ファイルが、どこから呼ばれ、何に依存しているかを確認する。
  影響範囲を理解してから手を動かす。検索(grep等)で使用箇所を洗い出すと良い。
- **既存のテストがあるか確認する。** `package.json` の scripts、`tests/` や `__tests__/`、`*.test.*`、
  `pytest`/`unittest` 等を探す。あれば**変更前に一度実行し、現状がパスすることを確認**しておく
  (これが「変更前のベースライン」になる)。

### 2. 変更中:壊さない編集を心がける

- **依頼された範囲だけ変更する。** 関係ないリファクタや整形を勝手に混ぜない。
- **既存の関数シグネチャや戻り値の形を安易に変えない。** 変えるなら呼び出し側を全て追って直す。
- **削除より追加・分岐を優先する。** 既存コードを消す前に、本当に使われていないか使用箇所を確認する。

### 3. 変更後:回帰していないか確認する(最重要)

直した箇所が動くのは当たり前。**「直していない既存機能が壊れていないか」を確認するのが回帰防止の本体。**

- **テストがある場合:** 変更後にもう一度テストを実行し、**変更前にパスしていたものが全てパスし続けているか**
  確認する。新たに失敗が出たら、それが回帰。原因を特定して直すか、変更を見直す。
- **テストがない場合:** 変更箇所に関連する主要な動作を手動で確認する。最低限、
  「直した機能」+「その機能と同じ画面・同じモジュールにある既存機能」が今までどおり動くかを確かめる。
  確認できない場合は、ユーザーに「ここは自動で確認できないので、◯◯の動作を試してほしい」と具体的に伝える。
- **`git diff` で意図しない変更が混ざっていないか最終確認する。** コミット前に必ず差分を見る。
  想定外のファイルや行が変わっていたら立ち止まる。

### 4. テストが無いプロジェクトへの提案

テストが一切ない場合、回帰は起きやすい。重要な機能を触ったときは、
**その機能に対する簡単なテストを1つ追加することをユーザーに提案してよい**(必須ではない)。
小さなテストでも、次回以降の回帰検知に大きく効く。

## 作業開始時の判断フロー(Git)

変更を始める前に、まず「この変更は壊れたら困るか?」を判断する。

### A. 軽微な変更 → `main` で直接作業

タイポ修正、コメントや文言の変更、README やドキュメントの更新、設定値の小さな調整など、
**失敗してもほぼ何も困らない変更**は、ブランチを作らず `main` で直接編集してよい。

1. `main` にいてクリーンな状態か確認する(`git status` / `git branch --show-current`)
2. 編集する
3. `git diff` で差分を確認する
4. コミットする(下記「コミットの作法」参照)

### B. 壊れたら困る変更 → ブランチを切って作業

新機能の実装、既存ロジックの変更、複数ファイルにまたがる修正、リファクタリング、依存関係の変更など、
**失敗すると今動いている状態が壊れる可能性がある変更**は、必ず作業用ブランチを切る。

判断に迷ったら B(ブランチ)を選ぶ。ブランチを切るコストは小さく、`main` を壊すコストは大きい。

1. `main` が最新かつクリーンな状態か確認する(`git status`)
2. 作業用ブランチを作って切り替える
   ```bash
   git switch -c feature/<わかりやすい名前>
   ```
   命名例: `feature/login-form`, `fix/null-crash`, `refactor/api-client`
3. そのブランチ内で編集し、こまめにコミットする
4. **回帰防止の手順(上記)に従って動作・テストを確認する**
5. 問題なければ `main` にマージする(下記「マージの作法」参照)

## コミットの作法

- **意味のあるまとまりごと**にコミットする。「動く状態になった」「一機能できた」などが区切りの目安。
- **動く状態でコミットする。** 壊れた状態のままコミットを重ねない(戻り先として使えなくなる)。
- 一度に大量の無関係な変更を 1 コミットに詰め込まない。後から戻りにくくなる。
- コミット前に必ず `git diff`(または `git status`)で内容を確認する。

```bash
git add -A
git commit -m "<変更内容を簡潔に>"
```

メッセージ例:
- `feat: ログインフォームを追加`
- `fix: 未入力時のクラッシュを修正`
- `refactor: API呼び出しを共通関数に整理`
- `docs: READMEのセットアップ手順を更新`

## 回帰してしまったときの戻し方

回帰に気づいたら、無理に上から直そうとせず、**直前の動く状態に戻る**のが安全。

```bash
# 直近のコミット以降の変更を全て破棄して、最後にコミットした状態に戻す
git restore .

# 特定ファイルだけ元に戻す
git restore <ファイル名>

# コミットしてしまった後で、その1つ前の状態に戻したい(変更内容は作業ツリーに残す)
git reset --soft HEAD~1
```

ブランチで作業していた場合は、最悪そのブランチを捨てれば `main` は無傷:
```bash
git switch main
git branch -D feature/<名前>
```

履歴を書き換える `reset --hard` や `push --force` は、ユーザーの明示的な確認なしには行わない。

## マージの作法(状況で判断)

ブランチでの作業が完了し、**回帰がないことを確認できたら** `main` に取り込む。
**マージ方法は状況に応じて選んでよい。** 迷ったら通常の `merge` を選ぶ。

```bash
git switch main
git merge feature/<名前>
```

判断の目安:
- **通常の `merge`(デフォルト)** — 基本はこれでよい。確実で安全。
- **`rebase` してから merge** — コミット履歴を直線的に綺麗に保ちたいとき、かつブランチがまだ Push されていないとき。
  すでに Push 済みのブランチを rebase すると履歴が食い違うので避ける。
- どちらが適切か不明なときは通常の `merge` にする。

マージ後、不要になったブランチは削除してよい:
```bash
git branch -d feature/<名前>
```

## Push の作法(作業の切れ目ごと)

Push は「リモート(GitHub 等)へのバックアップ・共有」のために行う。**毎コミットごとには行わない。**
**作業の切れ目** — 一つの機能が完成した、一区切りついた、作業を中断する — のタイミングで Push する。

```bash
# main を Push
git push

# 新しいブランチを初めて Push するとき
git push -u origin feature/<名前>
```

リモートが設定されていない場合は Push せず、ローカルのコミットだけで進めてよい。
その場合はユーザーに「リモートが未設定なのでローカルのみで進めています」と一言伝える。

## やらないこと(一人開発では不要)

以下はチーム開発向けの仕組みなので、このワークフローでは行わない:

- Pull Request の作成(レビュアーがいないため。ブランチから直接マージしてよい)
- `develop` / `release` / `hotfix` などの複雑なブランチ運用(`main` + 作業ブランチで十分)
- アクセス権限や保護ルールの設定

## 安全に関する注意

- ユーザーの明示的な確認なしに、`push --force`、`reset --hard`、履歴の書き換え(共有済みブランチの rebase)、
  `main` への破壊的操作は行わない。
- コミットされていない変更があるブランチを削除する前に、必ず内容を確認する。
- 大きな変更を `main` で直接始めようとしていることに気づいたら、いったん止めてブランチを切る。
- 確認(テスト・動作チェック)を飛ばしてマージ・Push しようとしていることに気づいたら、いったん止める。
